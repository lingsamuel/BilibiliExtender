# Repository Guidelines

工作语言为中文：使用中文交互、编写文档或者注释。

项目概述在文件`README.md`中。
你必须遵守编码指南文件：`CODE_GUIDE.md`中的内容。

## 代码组织：工程化

你编写的代码应该工程化，不能把所有代码都塞进一个大文件里。

## 开发模式：Spec Drive Development

你必须及时编写并更新功能模块的 Spec 文档，然后依照文档的指示编写代码。

文档需要放置在 docs 目录下。

## 更新文档

如果需要添加或修改大型功能或者核心运算逻辑，你需要相应的更改文档。
对于大型更新或者核心逻辑更新，你不能先写代码，而是与我确认文档。
我批准后，你可以更新对应的文档，然后再次确认文档是否正确更新。
再次批准后，你可以开始写代码。

不过，对于小型更新（如：样式更新、不涉及核心逻辑等），你可以忽略此规则。
或者，当你申请写文档或者申请批准文档时，如果我明确在回复中指出这次的修改可以不改文档，也可以忽略此规则。

## 必须确认需求

对于任何需求，你不能直接编写代码，而是要遵守下面的规则。

### 新功能需求

你首先需要分析需求，向我确认任何可能的模糊点。你不能猜测我的需求，对于任何不清楚的地方都要向我询问，直到一切都清晰、定义良好。

然后你需要更新或者新增文档，在我确认后，才能开始执行。

### 错误修复（Bugfix）需求

你首先需要仔细分析相关的代码，并向我解释。如果可能还需要向我询问错误的现象（难以或者无法复现的Bug除外），以精准定位错误所在。

在分析出真正的错误后，视其复杂程度来决定是否要记录或者更新文档。然后才能开始执行。

## 添加注释

你必须为关键或者复杂的逻辑添加中文注释。关键的函数、尤其是公共 API 必须要添加注释，写明（如存在）使用场景、前置要求等。

但是你的注释不能是函数名或者语句的字面翻译，需要深入解释复杂逻辑和实现原理。
