# 标签页触发机会式刷新 Spec

## 1. 背景

当前调度器的常规刷新主要依赖 alarm 与“过期后收集任务”。这会带来两个问题：

1. 同一批作者/收藏夹容易在接近时间被初始化，后续又在接近时间一起过期，形成稳定的批量刷新相位。
2. routine 刷新与用户真实访问时机脱节，容易在某个时间窗内集中发起多次请求，增加风控风险。

本功能尝试利用“用户真实打开 B 站标签页”的自然时机，顺手执行一小部分低峰值 routine 刷新，用更平缓的方式更新数据，降低后续 Burst 补齐与大批量常规刷新的概率。

## 2. 目标

### 2.1 核心目标

- 在用户打开 B 站页面时，机会式刷新少量 routine 数据。
- 控制单位时间内的真实 HTTP 请求数，避免短时间 burst。
- 优先降低“数据过旧导致后续集中补刷”的风险。
- 不改变手动刷新、缺缓存补齐、分页补页等优先路径“尽快执行”的原则。

### 2.2 非目标

- 不追求严格按秒均匀发请求。
- 不依赖后台常驻或长时间 sleep。
- 不劫持或复用页面自身的网络响应。
- 不替代初始化、异常恢复、低频兜底所需的全量刷新链路。

## 3. 术语

- **机会式刷新（Opportunistic Refresh）**：由“初次打开匹配站点标签页”触发的低峰值 routine 刷新。
- **请求预算（Request Budget）**：一次机会式刷新允许消耗的真实 HTTP 请求数量，而非逻辑任务数量。
- **全局防抖（Global Debounce）**：避免用户短时间连续打开多个标签页时触发多轮机会式刷新。
- **滑动窗口限流（Sliding Window Limit）**：统计过去一段时间内机会式刷新实际消耗的请求数，并限制上限。
- **实体冷却（Entity Cooldown）**：同一个作者、同一个收藏夹、收藏夹列表在最小刷新间隔内不得重复进入机会式刷新。

## 4. 功能概述

### 4.1 触发器

在扩展已注入的 B 站页面中，content script 于匹配页面的每个标签页首次加载完成后，向 background 上报一次“标签页初次打开”事件。

首版约束：

- 仅将“每个标签页首次加载”视为触发器。
- 不将切换标签页、聚焦恢复、滚动、轮询心跳作为首版触发器。
- 后续若增加更细粒度的激活信号，必须复用同一套防抖与预算控制。

### 4.2 触发前置条件

机会式刷新只在以下条件全部满足时才允许执行：

1. 当前不存在 Burst 队列运行或待执行任务。
2. 当前不存在手动刷新、缺缓存补齐、分页补页等优先任务的执行压力。
3. 当前不处于全局冷却（如 WBI 风控冷却）中。
4. 命中全局防抖要求。
5. 命中滑动窗口限流要求。

任一条件不满足时，本次标签页事件直接跳过，不排队补偿。

## 5. 限流与节流设计

### 5.1 预算口径

预算必须按真实 HTTP 请求数计算，不能按“作者数 / 收藏夹数 / 任务数”计算。

原因：

- 作者投稿刷新通常至少涉及投稿接口，且常伴随 Card 刷新。
- 收藏夹列表同步通常涉及当前用户信息与收藏夹列表请求。
- 收藏夹内容刷新可能展开为多页请求，成本波动明显。

因此机会式刷新调度器需要维护“预估成本”与“实际消耗”两个口径：

- **预估成本**：决定是否允许某个候选任务进入本轮机会式刷新。
- **实际消耗**：用于滑动窗口限流统计。

### 5.2 全局防抖

必须存在一次全局时间防抖，防止用户短时间连续打开多个 B 站标签页时触发多轮机会式刷新。

设计要求：

- 防抖作用域为整个扩展，而非单个标签页。
- 在防抖窗口内，后续“标签页初次打开”事件直接忽略。
- 防抖时间为独立参数，具体默认值待实现前最终确认。

### 5.3 滑动窗口限流

过去 30 分钟内，机会式刷新实际消耗的 HTTP 请求数不得超过 60。

首版建议实现为“滑动窗口日志”：

1. 记录每次机会式刷新发出的实际请求时间戳。
2. 每次准备触发前，先清理 30 分钟之前的记录。
3. 若当前窗口内记录数已达到上限，则本次直接跳过。

说明：

- 这里限制的是“机会式刷新额外带来的请求”，不包含手动刷新、缺缓存补齐、分页补页等优先路径。
- 若某个机会式刷新任务的实际请求数超出预估，应按真实发出的请求逐条记账。

### 5.4 实体冷却

下列实体默认 10 分钟内最多进入机会式刷新 1 次：

- 单个作者投稿缓存机会刷新
- 单个收藏夹内容刷新
- 收藏夹列表同步（全局单例）

此外还需一条机会式刷新总开关：

- 当候选集中“最老实体”也未超过最小刷新阈值时，本次不发起任何请求。

## 6. 机会式刷新范围

### 6.1 纳入范围

首版记录以下三类数据源：

1. 已追踪作者的投稿缓存刷新（头部刷新 + 连续窗口维护）
2. 收藏夹列表同步（“我的收藏夹列表”）
3. 收藏夹内容刷新（用于更新收藏夹中包含的作者列表）

### 6.2 优先级建议

尽管三类任务都可纳入机会式刷新，但其成本与收益不同，推荐优先级如下：

1. 作者投稿缓存刷新
2. 收藏夹列表同步
3. 收藏夹内容刷新

说明：

- 作者投稿缓存刷新成本相对稳定，且最直接影响前台可见数据新鲜度。
- 收藏夹列表同步是全局单例任务，便于控制频率。
- 收藏夹内容刷新成本波动大，应在预算允许时才执行。

## 7. 作者投稿刷新设计

### 7.1 当前缓存模型

当前作者投稿缓存的真相已经不是旧的共享页切片模型，而是：

1. **连续窗口（`continuousVideos`）**：
   - 表示“当前能够证明从头部开始连续成立”的一段投稿序列；
   - 时间流与默认作者列表都优先消费这段数据。
2. **精确页块（`blocks`）**：
   - 表示任意 `pn/ps` 请求落成的精确页结果；
   - 手动翻页、当前页刷新、连续窗口补块都依赖页块缓存；
   - 非第一页的“按作者分页”必须读取精确页块，不能再按旧 `sourcePn` 近似切片。

因此，机会式刷新不能再沿用旧的“按页同步边界”设计，而必须服务于当前真实缓存模型。

### 7.2 机会式作者刷新目标

在当前模型下，作者投稿的机会式刷新目标应定义为：

1. 刷新作者头部，更新最新版本摘要与连续窗口起点。
2. 在预算允许时，顺手维护少量连续窗口补块，尽量让连续窗口覆盖到目标长度。
3. 不主动替代用户翻页语义，不把机会式刷新结果包装成“某次手动翻页返回的精确页”。

换句话说，机会式作者刷新关注的是“连续窗口维护”，而不是“把页码一直往后推”。

### 7.3 机会式作者任务类型

首版作者侧仅考虑两类机会式任务：

1. **头部刷新**
   - 固定请求作者投稿第 1 页；
   - 同步刷新 Card；
   - 写入新的首页块，并立即尝试重建连续窗口。
2. **连续窗口维护**
   - 当某作者当前连续窗口长度低于目标窗口长度时，补取下一块精确页；
   - 该任务默认低于“头部刷新”优先级；
   - 其目标是扩展 `continuousVideos`，不是为前台手动翻页预热任意页块。

不纳入机会式刷新的任务：

- `REQUEST_AUTHOR_PAGE`
- `refresh-author-current-page`
- 任何显式服务于前台用户翻页等待语义的精确页请求

这些任务仍保持用户驱动、Burst 优先、精确页块返回的独立语义。

### 7.4 头部刷新语义

单个作者的机会式头部刷新在当前代码语义下应当等价于：

1. 请求 `pn=1` 的作者投稿块；
2. 刷新当前作者 Card 信息；
3. 更新最新版本摘要（总数 + tagCounts）；
4. 以新首页块为起点，尝试与现有块池、旧连续窗口重建更长的 `continuousVideos`。

因此，机会式头部刷新并不需要另起一套 `frontierPubdate` 元数据；当前缓存中的版本摘要、首页块与连续窗口就是作者侧的同步语义基础。

### 7.5 连续窗口维护语义

机会式连续窗口维护的判断标准应与当前调度器保持一致：

1. 第二页仍是默认优先的防御性目标；
2. 若当前连续窗口长度不足以覆盖 `2 + authorContinuousExtraPageCount` 页窗口，则允许补取后续精确页块；
3. 每次补块后都必须重新尝试连续窗口重建，而不是简单追加页码。

说明：

- 连续窗口维护属于“头部刷新之后的可选加分项”，不是每次标签页打开都必须执行。
- 若请求预算不足，只做头部刷新即可；补块任务可以留给下次机会式刷新或常规调度。

### 7.6 与精确分页块的边界

当前作者缓存模型已经明确区分：

1. `continuousVideos`：供时间流与默认作者列表消费的连续真相；
2. `blocks`：供精确分页读取的页块真相。

因此，机会式刷新必须遵守以下边界：

1. 机会式刷新可以生成或更新精确页块，但其主要目标仍是重建连续窗口。
2. 机会式刷新不得把“刚好命中的某个页块”当成用户手动翻页结果回传。
3. `REQUEST_AUTHOR_PAGE` 仍需通过 `requestedAt -> fetchedAt` 校验来确认“这是本次用户请求之后的新页块”。

### 7.7 页数与预算上限

作者侧机会式刷新必须受请求预算约束，避免单次标签页打开退化成大量补块。

首版建议：

1. 每次标签页触发中，作者侧优先消费头部刷新预算。
2. 若头部刷新后仍有剩余预算，且连续窗口明显不足，再允许补取极少量后续页块。
3. 单个作者在一次机会式刷新中最多额外补取 1 到 2 个连续窗口页块；具体默认值待实现前确认。

### 7.8 为什么不再采用旧的作者侧时间交叉增量设计

此前讨论过把作者投稿刷新设计成“基于时间交叉的页级增量同步”。在旧的共享页切片模型下，这个方向是合理的。

但在当前实现中：

1. 作者缓存主模型已切换为“连续窗口 + 精确页块”；
2. 手动翻页不再消费共享页切片，而是依赖精确页块；
3. 连续窗口的正确性主要由块拼接与版本摘要保证，而不是单独的页边界水位。

因此，机会式作者刷新应直接建立在当前真实缓存模型上，而不是再并行引入一套旧式 `frontierPubdate` 同步机制。

## 8. 收藏夹内容刷新设计

### 8.1 现状问题

当前实现的收藏夹内容刷新会从第 1 页开始全量翻页，直到 `has_more=false` 或命中安全上限。

这对于“大收藏夹”来说成本过高，不适合作为高频机会式刷新路径。

### 8.2 增量刷新原则

机会式刷新中的收藏夹内容同步默认不是“初始化”，而是“更新”。因此可以采用基于 `fav_time` 的增量刷新策略。

前提：

1. 收藏夹列表接口按收藏时间降序返回。
2. 实现中应显式要求按收藏时间排序（如 `order=mtime`），不能依赖默认行为。

### 8.3 fav_time 水位

增量刷新以 `fav_time` 作为边界水位，用于判断“是否已经追上旧数据”。

但需要区分两个概念：

- **headFavTime**：当前列表头部最新项的 `fav_time`
- **frontierFavTime**：上次同步结束时，已同步窗口最老一条数据的 `fav_time`

其中真正用于判断“是否已经追上旧数据”的，是 `frontierFavTime`，而不是 `headFavTime`。

说明：

- 若只是判断“这次增量是否已经追上旧边界”，`fav_time` 水位即可成立。
- 若需要识别“已对齐窗口内哪些具体条目被删除”，则仅有 `fav_time` 不够，仍需保留少量条目身份快照。

### 8.4 需要新增的收藏夹同步元数据

为支持机会式增量刷新，每个收藏夹至少需要新增以下同步元数据：

- `frontierFavTime`：上次同步窗口最老项的 `fav_time`
- `syncedItemCount`：上次同步覆盖的条目数
- `syncedTailPage`：上次同步结束时所处页号
- `totalCountAtLastSync`：上次同步时接口返回的总条数
- `frontierItems`：上次同步前沿窗口的少量有序条目快照。每项至少包含稳定标识与 `fav_time`
- `lastIncrementalRefreshAt`：最近一次机会式增量刷新时间
- `lastFullRefreshAt`：最近一次全量重建时间

其中：

- `frontierFavTime` 用于边界判断。
- `frontierItems` 用于在“已追上旧边界”后，对重叠窗口执行局部 diff，并识别一部分删除事件。

### 8.5 增量刷新流程

#### 8.5.1 快路径

1. 先请求第 1 页。
2. 若第 1 页已触及或跨过 `frontierFavTime`，则多数情况下可认为已追上旧边界。
3. 若本次扫描窗口与 `frontierItems` 对齐，则可在重叠窗口内执行局部 diff：
   - 识别新增条目
   - 识别已对齐窗口内的局部删除条目
4. 完成局部 diff 后，再基于当前已拿到的内容更新收藏夹作者列表，并刷新同步元数据。

#### 8.5.2 边界页估算

若第 1 页未触及 `frontierFavTime`，说明新增收藏较多，需要进一步探测旧边界位置。

可利用本次第 1 页响应中的总条数，与 `totalCountAtLastSync` 做差，估算上次同步边界在当前列表中的大致页号。

例如：

- 上次同步结束在第 5 页附近
- 本次前端新增了若干条目
- 则旧边界可能被整体向更深页推移

此时可以直接跳到估算页附近进行探测，而不是从第 2 页顺序线性扫到边界。

#### 8.5.3 边界探测方向

不建议把方向固定写死成“从第 N 页反向同步 N、N-1、N-2……”。更稳妥的方式是：

1. 先探测估算页。
2. 比较该页 `fav_time` 区间与 `frontierFavTime` 的关系。
3. 再决定继续向更深页还是更浅页探测。

判断口径：

- 若该页最老 `fav_time` 仍新于 `frontierFavTime`，说明旧边界还在更深页。
- 若该页最新 `fav_time` 已老于 `frontierFavTime`，说明旧边界更靠近前面页。
- 若 `frontierFavTime` 落在该页时间区间内，则说明边界已逼近，可结束探测。

### 8.5.4 已对齐窗口内的删除识别

机会式增量刷新除了发现新增外，也应尽量识别一部分删除事件。

示例：

- 旧窗口：`a b c d e`
- 新窗口：`a b d e`

若本次扫描已经覆盖到旧窗口边界，并且新旧窗口顺序仍可对齐，则可以安全推断 `c` 已被删除。

为支持这类能力，需要满足以下条件：

1. 当前扫描窗口已追上 `frontierFavTime`，即旧边界已经进入本次可见范围。
2. 当前扫描窗口与 `frontierItems` 存在可对齐的重叠部分。
3. 被判定删除的条目位于“已对齐的重叠窗口”内，而不是仅仅没有出现在更浅的页中。

实现原则：

- `fav_time` 负责判断边界是否追上。
- `frontierItems` 负责在已对齐窗口内定位“具体是谁被删了”。
- 不允许仅凭“本页没看到某条旧数据”就判定其删除；必须证明这条数据原本应当出现在本次已对齐窗口中。

说明：

- 该能力只能识别“已对齐窗口内的局部删除”，不能保证发现更深层、未被本次扫描覆盖的删除。
- 若增量扫描未追上旧边界，则本次不得进行删除推断。

### 8.6 页数上限

机会式增量刷新必须限制最大探测页数，避免退化成全量扫描。

首版讨论结论：

- 每次机会式刷新中，收藏夹内容最多探测 3 页。
- 若 3 页内仍未追上旧边界，则本次记为“增量未完成”。
- 是否在后续机会式刷新继续追，或回退到低频全量重建，交由后续策略决定。

### 8.7 全量重建保底

基于 `fav_time` 的增量刷新不能完全替代全量重建，必须保留以下兜底路径：

1. 初始化时的首次全量构建
2. 增量刷新连续多次未追上旧边界
3. 接口排序口径异常、前沿窗口无法对齐，或元数据缺失
4. 低频全量校验（周期待实现前确认）

## 9. 机会式刷新与现有调度器的关系

### 9.1 语义定位

机会式刷新属于 routine 刷新，不属于优先任务。

因此：

- 不得抢占手动刷新
- 不得抢占缺缓存补齐
- 不得抢占分页补页
- 不得在 Burst 运行期间继续发起请求

### 9.2 建议的对外语义

可为调度器新增一类独立语义，例如：

- `trigger: 'tab-open-opportunistic'`
- `mode: 'opportunistic'`

以便在调试面板和历史记录中与 alarm routine、manual refresh、Burst 明确区分。

### 9.3 调度原则

机会式刷新不要求“一次触发就把预算花满”。更合适的原则是：

1. 在预算允许范围内，按优先级挑选候选任务。
2. 遇到候选任务成本过高时可直接跳过，避免单次触发变成 burst。
3. 若本轮没有合适候选，则本次机会刷新直接结束。

## 10. 首版建议范围

为了降低实现与风控风险，首版建议遵循以下范围：

1. 仅使用“标签页初次打开”作为触发器。
2. 启用全局防抖。
3. 启用 30 分钟 / 60 请求的滑动窗口限流。
4. 单个作者、单个收藏夹、收藏夹列表默认 10 分钟冷却。
5. 作者投稿缓存机会刷新纳入首版，默认以“头部刷新”为主、连续窗口维护为辅。
6. 收藏夹列表同步纳入首版。
7. 收藏夹内容刷新采用 `fav_time` 增量同步，但限定最大探测页数。
8. 初始化与异常场景继续保留全量重建路径。

## 11. 待确认项

以下参数已达成方向共识，但实现前仍需最终确认默认值或细节：

1. 全局防抖窗口时长
2. 单次机会式刷新的请求预算默认值
3. 单个作者在一次机会式刷新中允许补取多少个连续窗口页块
4. 收藏夹内容增量刷新“最大探测页数”是否固定为 3
5. 增量未完成时的后续策略：
   - 下次继续机会式刷新补追
   - 或低频回退到全量重建
6. 全量重建的兜底周期是否需要显式配置
