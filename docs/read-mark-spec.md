# 已阅标记与观看状态 Spec

## 1. 概述

为"按作者"和"时间流"模式添加已阅标记功能，解决固定显示数量导致旧视频过多、新视频显示不全的问题。同时基于投稿接口返回的播放进度，展示视频的查看/观看状态。

## 2. 术语

- **已阅时间点（ReadMark）**：用户手动标记的时间戳，表示"此时间点之前的视频我已经看过了"。按分组（`groupId`）独立记录，不跨分组共享。
- **已阅边界（ReadBoundary）**：用于区分“未读/已阅”的时间分界线。时间流按日设置时，边界固定为“目标日期次日 00:00:00（本地时区）”。
- **点击记录（Clicked）**：用户在本扩展中点击过某视频链接。
- **播放进度（PlaybackPosition）**：投稿接口字段 `playback_position`，范围 `0-100`，表示观看进度百分比。
- **已查看（Viewed）**：满足“点击记录命中”或“播放进度 >= 10”。
- **已看完（Finished）**：播放进度 `>= 90`。

## 3. 功能需求

### 3.1 已阅标记

#### 3.1.1 数据模型

```ts
// 每个分组的已阅记录，按 groupId 索引，不跨分组共享
interface GroupReadMark {
  groupId: string;
  // 最多保留 10 条，按时间倒序排列（最新在前）
  // 每条记录为 Unix 时间戳（秒），表示该时间点及之前的视频已阅
  timestamps: number[];
}

// 存储键：bbe:group-read-marks
// 存储格式：Record<string, GroupReadMark>（groupId -> GroupReadMark）
```

#### 3.1.2 "标记已阅"按钮

- 位置：工具栏区域，与"手动刷新"、"关闭"并列。
- 行为：点击后，仅向当前分组的 `timestamps` 追加“当前时间戳（秒）”。
- 每个分组最多保留 10 条记录，超出时丢弃最旧的。
- 按钮文案："标记已阅"。

#### 3.1.3 已阅时间点切换

- 位置：工具栏区域，"标记已阅"按钮旁边。
- 形式：下拉菜单（`<select>`）。
- 选项顺序固定：
  - "全部"——不过滤，显示所有视频（等同于当前行为）；该分组红点固定为 `0`。
  - "`N天内`"——默认已阅区间（由 `defaultReadMarkDays` 推导；原“`N天前`”文案统一改为“`N天内`”）。
  - 各已阅时间点——来自当前分组自己的 `timestamps`，按时间倒序排列，格式化为可读日期时间。
- 默认选中：当前分组最新的已阅时间点；若当前分组无历史记录，则默认选中 "`N天内`"。

#### 3.1.4 视频过滤逻辑

选中某个分组已阅时间点 `T` 后：

- 有效边界 `Tb` 定义：
  - `T = 0`（全部）：不过滤；
  - `T = -1`（`N天内`）：`Tb = graceTs`；
  - `T > 0`：`Tb = T`。

**按作者模式**（`T !== 0`）：
- 对当前分组的每位作者统一使用同一个 `Tb`；
- 显示 `pubdate >= Tb` 的所有视频；
- 额外显示 `pubdate < Tb` 的最新 N 个视频（作为上下文参考），N 由设置项 `extraOlderVideoCount` 控制，默认 1。

**时间流模式**（`T !== 0`）：
- 不使用 `extraOlderVideoCount` 规则，避免把“已阅前补充视频”带入时间流造成时间跨度异常。
- 默认按统一边界过滤后混合所有作者并按时间排序；支持“滚动触底 + 手动加载更多按钮”双触发继续加载更早视频。
- “加载更多”按钮固定显示在时间流底部；即使当前没有更多数据也不隐藏（点击后可提示无更多）。
- 第二页仅为预取加速项，不保证已存在；构造器在命中边界时可将“缺失第二页”直接升级为 Burst 高优先级补页。
- 时间流 UI 按“有投稿日期”分段，并显示左侧时间轴。
- 时间轴文案：`今天` / `昨天` 保持相对文案；距离今天 2–7 天（含）显示 `N 天前`；更早日期显示 `MM-DD`。
- 时间轴压缩窗口：可视区域内日期节点全部显示；可视区域外仅保留上下各 2 个“有投稿日期”节点，离屏节点退化为小黑点并停靠在顶部/底部区域。

#### 3.1.5 时间流按日设置已阅（新增）

- 仅在“时间流模式”提供该交互；按作者模式不提供按日设置入口。
- 时间轴“日期圆圈”可交互：
  - hover 圆圈显示半透明粉色对勾；
  - 点击圆圈后，仅向当前分组写入新的已阅记录，并立刻将当前分组 `selectedReadMarkTs` 切换到这条新记录；
  - 激活态为“对勾不透明 + 圆圈边框为粉色（不填充）”。
- 圆圈对应的按日边界时间戳定义：
  - 选中日期 `D` 时，写入 `D+1` 日 `00:00:00`（本地时区）作为 ReadBoundary；
  - 语义为“`D` 及更早已阅，`D+1` 起视为未读”。
- 圆点对勾模式的时间流连续性补充：
  - 若该 ReadBoundary 落在当前 `N天内` 窗口中，则时间流仍渲染 `N天内` 的全部内容（不仅是 `D+1` 之后）。
  - 无论是否落在 `N天内` 窗口中，时间流都必须保留 `D` 日投稿内容，保证分隔符两侧语义连续。
- 激活态自动回显：
  - 若当前分组的有效已阅时间点已落在某个日期 `D` 对应区间，也需自动给该日期圆圈应用激活样式。
  - 区间判定统一使用 ReadBoundary 映射，不依赖是否由圆圈点击写入。
- 日期文字按钮继续承担“滚动跳转到该日”的功能，不改变其现有职责。

#### 3.1.6 “上次看到这里”分隔符（新增）

- 时间流中在“未读/已阅”分界处插入粉色分隔符。
- 组成：粉色分隔线 + 粉色文案“上次看到这里” + 向下箭头。
- 方向语义：分隔符上方是未读更新，下方是已阅内容。
- 布局要求：分隔符必须额外占据一行高度，作为独立块级行渲染，不能挤占或叠加到既有 `bbe-mixed-day-section` 内容区里。

#### 3.1.7 按作者模式边界竖线（新增）

- 按作者模式中，在每位作者的 VideoCard 列表内，于“已阅边界前后”两张卡片之间插入粉色竖线分割标记。
- 竖线只用于指示边界，不应改变卡片网格布局（不可导致卡片尺寸或列数变化）。
- 该分割逻辑必须支持任意时间戳边界（不依赖按日分段）。
- 若该作者当前可见视频全部早于已阅边界（即全部由 `extraOlderVideoCount` 补充展示），则不显示竖线分割。

### 3.2 已查看标记（点击追踪）

#### 3.2.1 数据模型

```ts
// 存储键：bbe:clicked-videos
// 存储格式：Record<string, number>（bvid -> 点击时间戳）
// 点击记录不按时间过期；仅当 bvid 不再存在于任何作者视频缓存中时才清理
```

#### 3.2.2 行为

- 监听 VideoCard 组件的点击事件（卡片打开视频）。
- 点击时将 `bvid` 和当前时间戳写入存储。
- UI 判定规则：点击记录命中后可显示"已查看"标签；若同一卡片满足"已看完"，则不显示"已查看"。
- 点击记录在语义上等同布尔“已点击”，时间戳仅用于兼容历史结构与调试。

### 3.3 播放进度派生状态

#### 3.3.1 数据获取

- 数据来源：作者投稿接口 `/x/space/wbi/arc/search` 返回的 `playback_position` 字段。
- 不额外请求观看历史接口，不引入额外缓存层。

#### 3.3.2 状态判定规则

```ts
// playback_position 范围 0-100（百分比）
if (playback_position >= 90) {
  status = 'finished'; // 已看完
} else if (playback_position >= 10) {
  status = 'viewed'; // 自动视作已查看
} else {
  status = 'unviewed';
}
```

#### 3.3.3 展示

- VideoCard 底部显示观看进度条（`playback_position > 0` 时显示）。
- `playback_position >= 90`：显示"已看完"标签。
- `10 <= playback_position < 90`：可显示"已查看"标签（即使没有点击记录）。
- 若已看完，则隐藏"已查看"标签，避免语义重复。

### 3.4 设置项变更

原设置项 `authorPerCreatorCount`（作者模式每作者展示量，默认 10）不再适用于新的已阅过滤逻辑，改为：

- **`extraOlderVideoCount`**：已阅时间点之前额外显示的视频数量，默认 `1`。
  - 含义：选中已阅时间点后，每位作者除了显示已阅之后的新视频外，还额外显示已阅之前最新的 N 个视频作为上下文。
  - 设置页标签："已阅前额外显示数量"。

原设置项 `mixedInitialTargetCount`（混合模式初始目标量，默认 50）重命名为：

- **`timelineMixedMaxCount`**：时间流模式最大加载数量，默认 `50`。
  - 含义：时间流模式下最多加载的视频总数上限。新的已阅过滤行为下，此设置用于防止某个分组加载过大量数据。
  - 设置页标签："时间流模式最大加载数量"。

注意：当用户在已阅下拉菜单中选择"全部"时，不做任何过滤，显示所有已拉取的视频（行为与改动前一致）。

### 3.5 红点与未读数规则

#### 3.5.1 分组红点（抽屉侧边栏）

- 分组红点与“该分组当前已阅时间点”绑定，不再因“打开分组/加载内容”自动清除。
- 若分组 `savedReadMarkTs === 0`（全部），该分组未读数固定为 `0`。
- 若分组 `savedReadMarkTs > 0`，分组基线为 `savedReadMarkTs`。
- 若分组 `savedReadMarkTs === -1`，分组基线回退到 `graceTs`（由 `defaultReadMarkDays` 推导）。
- 分组未读数 = 分组内去重后视频中同时满足：
  - `pubdate > groupBaselineTs`
  - 且未命中“已查看（点击记录命中或 playback_position >= 10）”

#### 3.5.2 顶部入口 unread count（全局）

- 顶部入口 unread count 不等于“各分组未读数简单求和”，而是按作者聚合，避免跨分组重复与时间点冲突。
- 对每位作者：
  - 收集该作者所在的所有“非全部分组”（`savedReadMarkTs !== 0`）。
  - 对每个分组按 3.5.1 计算分组基线。
  - 取这些分组基线中的最大值作为该作者全局基线（“最新已阅时间点”）。
  - 若该作者只出现在“全部分组”中（没有任何非全部分组基线），则该作者全局未读数固定为 `0`。
- 全局 unread count 为所有作者未读视频数之和（按 `bvid` 去重），并同样扣除“已查看（点击记录命中或 playback_position >= 10）”的视频。
- “全部分组”规则：仅自身分组 unread 为 0，且不参与其他分组/作者的全局基线计算。

## 4. 存储

### 4.1 存储方案

使用 `chrome.storage.local`，在 manifest 中添加 `"unlimitedStorage"` 权限。

### 4.2 存储键

| 键 | 内容 | 说明 |
|---|---|---|
| `bbe:group-read-marks` | `Record<string, GroupReadMark>` | 按分组的已阅时间点 |
| `bbe:clicked-videos` | `Record<string, number>` | 点击过的视频 bvid -> 时间戳 |

### 4.3 清理策略

- `bbe:clicked-videos`：与作者视频缓存联动清理。仅当某 `bvid` 已不在任意作者缓存中时才删除点击记录。

### 4.4 旧数据处理

- 从“按作者已阅”切换到“按分组已阅”时，旧键 `bbe:author-read-marks` 直接丢弃，不做迁移。

### 6.3 时间流时间轴新增交互

```text
[日期文字按钮] [圆圈]
              ↑ hover: 半透明粉色对勾
              ↑ active: 不透明粉色对勾 + 粉色边框（不填充）
```

- 圆圈：设置已阅边界（写入 + 立即切换）。
- 日期文字：滚动跳转到对应日期分段。

### 6.4 时间流分隔符（独立行）

```text
──────────── 上次看到这里 ↓ ────────────
```

- 该行独立占位于时间流列表中，不覆盖现有日期分段内容。

### 6.5 时间流加载更多交互（更新）

- 列表底部常驻“加载更多”按钮，和触底自动加载并存。
- 二者复用同一 loadMore 流程，避免重复并发触发。
- 若加载链路触发了 Burst 补页且失败，显示错误提示，同时基于当前缓存 best-effort 返回。

### 6.6 按作者模式边界竖线

- 在某作者卡片列表内，边界两侧卡片之间渲染粉色竖线标记；
- 仅在“该作者既有边界之后视频，又有边界之前视频”时显示；
- 全部为边界之前视频（`hasOnlyExtraOlderVideos === true`）时不显示。

## 7. 实现要点

- 已阅时间点按分组 `groupId` 存储，不跨分组共享。
- `MARK_GROUP_READ_MARK` 支持可选 `readMarkTs`：
  - 不传时沿用“当前时间”语义；
  - 传入时使用指定秒级时间戳（用于时间流按日设置边界）。
- 过滤逻辑在 background 的 `toFeedResult` 中执行，content script 只负责展示。
- 视频“已查看/已看完”状态由 `playback_position` 与点击记录共同决定，不再依赖观看历史接口。
- 红点与全局 unread 计算会复用“已查看”判定，避免把已看过内容计入未读。
- 点击追踪在 content script 中直接写入 `chrome.storage.local`（content script 有权限）。
