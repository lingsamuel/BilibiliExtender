# Burst 通道拆分与预算状态机 Spec

> 状态说明：
> 本文描述的是 Burst 相关调度的下一步目标设计。
> 当前实现现状请先参考 `docs/scheduler-current-implementation.md`。
> 现有排序元数据与请求批次规则请继续参考 `docs/scheduler-request-ordering-spec.md`。

## 1. 背景

当前实现里，作者类 Burst 任务都落在同一个 `burst` 队列中：

- 用户手动翻页；
- 作者级刷新按钮；
- 无缓存作者首刷；
- `group-fav` 链出的无缓存作者补齐；
- 时间流边界补页。

这会带来两个问题：

1. 用户显式触发的单作者请求，会和后台大批量补缓存共享同一个 Burst 执行面。
2. 超大分组首次补缓存时，Burst 会持续排空，容易长时间占用作者接口预算。

本方案的目标不是改变“作者请求最终仍要串行执行”的事实，而是把 Burst 内部进一步拆层：

- 单作者手动请求独立出来；
- 批量手动请求与自动补缓存继续共享后端 ratelimit 视角下的 Burst 预算；
- 常规更新仍保持独立，不共享 Burst 的快慢冷却状态。

## 2. 目标

本方案希望达成：

1. 将 Burst 拆成三类逻辑通道：
   - `manual`
   - `manual-burst`
   - `auto-burst`
2. `manual-burst` 与 `auto-burst` 共享同一个 Burst 预算状态机。
3. 用户手动的单作者请求不受 Burst 预算和冷却影响。
4. 常规更新不共享 Burst 预算状态，只在两个 Burst 通道都空时运行。
5. 批量模式失败后保持阻塞，固定等待 60 秒后重试当前任务。
6. 调试页能明确显示当前 Burst 阶段、剩余预算、阻塞任务与最后错误。

## 3. 非目标

本方案当前不处理：

1. 不改点赞通道。
2. 不把 `group-fav` 与作者类 Burst 合并成统一队列。
3. 不把 Burst 预算改成“按真实请求数计费”。
4. 不把 `manual` 失败改成自动重试。
5. 不改变现有作者任务去重键 `(mid, pn, ps)`。

## 4. 通道模型

### 4.1 `manual`

`manual` 表示用户主动触发、且当前前台明确等待结果的非批量作者请求。

适用入口：

- “近期投稿”界面中的作者手动翻页；
- “全部投稿”界面中的作者手动翻页；
- 作者级刷新按钮。

约束：

1. 该通道内的请求被视为“单个任务”，不参与批量预算。
2. 该通道优先级最高。
3. 该通道不读取 Burst 的快慢冷却状态，也不消耗 Burst 预算。
4. 若请求失败，直接向前台返回失败通知，不自动重试。

### 4.2 `manual-burst`

`manual-burst` 表示用户主动触发、但属于分组级或批量级的作者刷新任务。

适用入口：

- 分组界面工具栏中的“刷新”；
- 设置页面中的“刷新投稿列表”；
- 上述入口在 `group-fav` 成功后衍生出的作者刷新任务。

说明：

1. 这类任务虽然由用户触发，但其后端压力模型属于批量作者请求。
2. 因此它不应与 `manual` 共享行为，而应与 `auto-burst` 共享 Burst 预算状态机。
3. `manual-burst` 优先级高于 `auto-burst`。

### 4.3 `auto-burst`

`auto-burst` 表示后台自动补缓存、自动补页等非用户显式等待结果的批量作者任务。

适用入口：

- `GET_GROUP_FEED` 发现无缓存作者后的首刷；
- `group-fav` 链出的无缓存作者补齐；
- 时间流边界补页；
- 其他未来的自动作者补缓存任务。

约束：

1. `auto-burst` 只有在 `manual-burst` 为空时才能开始执行。
2. `auto-burst` 与 `manual-burst` 共享同一个 Burst 预算状态机。

### 4.4 常规更新

常规更新包括现有：

- `author-video` 常规队列；
- `group-fav` 常规队列。

它们的约束保持为：

1. 不共享 Burst 的快慢冷却状态。
2. 不消耗 Burst 的任务预算。
3. 只在 `manual-burst` 与 `auto-burst` 都空时运行。
4. 若批量任务失败，同样采用“阻塞并固定等待 60 秒后重试当前任务”的错误策略。

## 5. 跨通道优先级

新的作者类执行优先级定义为：

```txt
manual
>
manual-burst
>
auto-burst
>
author-video / group-fav 常规更新
```

约束说明：

1. 高优先级只抢占“下一项执行权”，不强杀正在执行的当前任务。
2. `manual-burst` 与 `auto-burst` 之间共享同一个 Burst 状态机，但仍保留独立队列，以表达“用户手动批量高于后台自动补齐”的优先级。
3. `manual` 不参与 Burst 状态机，因此它的失败不应污染 `manual-burst / auto-burst` 的预算阶段。

## 6. Burst 共享状态机

### 6.1 设计原则

`manual-burst` 与 `auto-burst` 虽然逻辑上是两类不同来源的任务，但在作者投稿接口的后端 ratelimit 视角下，它们共享同一种压力来源。

因此：

1. 两个通道共享同一组配置项。
2. 两个通道共享同一个运行时状态：
   - 当前阶段；
   - 当前阶段已消耗任务数；
   - 下一次允许执行时间。

也就是说：

- 若 `manual-burst` 已消耗了 50 个“快速模式”任务预算；
- 切换到 `auto-burst` 执行时，`auto-burst` 只能继续使用剩余 10 个“快速模式”名额；
- 不能因为切换了通道就重置快速预算。

### 6.2 三阶段

Burst 共享状态机采用三阶段：

1. 快速阶段
   - 任务间隔：默认 `1s`
   - 预算：默认 `60` 个任务

2. 慢速阶段
   - 任务间隔：默认 `2s`
   - 预算：默认 `30` 个任务

3. 冷却阶段
   - 不执行 `manual-burst` / `auto-burst`
   - 默认等待 `2min`

### 6.3 阶段推进规则

规则如下：

1. 初始进入快速阶段。
2. 每成功或失败执行一个批量 Burst 任务，都记为一次任务消耗。
3. 快速阶段预算耗尽后，进入慢速阶段。
4. 慢速阶段预算耗尽后，进入冷却阶段。
5. 冷却阶段结束后，重置预算并回到快速阶段。

说明：

1. 本方案中的“预算”明确按任务数计算，而不是按真实请求数计算。
2. 本方案中的“间隔”明确按任务间隔计算，而不是按任务内部的单次 HTTP 请求间隔计算。

## 7. 错误处理

### 7.1 `manual`

`manual` 的错误语义：

1. 请求失败后立即结束该任务。
2. 直接向前台发送失败通知。
3. 前台沿用当前的报错提示方式（如右上角弹框）即可。
4. 不做自动重试。

### 7.2 批量模式

批量模式包括：

- `manual-burst`
- `auto-burst`
- `author-video`
- `group-fav`

这些模式统一采用阻塞式错误策略：

1. 任意错误都视为阻塞性错误。
2. 失败任务不跳过、不降级、不移出当前队首。
3. 固定等待 `60s` 后重试当前任务。
4. 直到该任务成功，后续同通道任务才可继续推进。

说明：

1. `WbiExpiredError` 仍是典型的阻塞性错误场景。
2. 本方案不区分“可恢复错误”和“不可恢复错误”，统一按阻塞重试处理。
3. 这意味着一个长期失败的任务可以长期卡住对应批量通道；这是该方案明确接受的调度语义，而不是 bug。

## 8. 配置项

设置页“高级设置”的 Burst 分组新增或调整：

```ts
interface BurstSettings {
  burstFastIntervalMs: number;      // 默认 1000
  burstFastBudgetTasks: number;     // 默认 60
  burstSlowIntervalMs: number;      // 默认 2000
  burstSlowBudgetTasks: number;     // 默认 30
  burstCooldownMs: number;          // 默认 120000
  burstErrorRetryMs: number;        // 默认 60000
}
```

规则：

1. `manual-burst` 与 `auto-burst` 共享这组配置。
2. `manual` 不读取这组配置。
3. `author-video` / `group-fav` 不读取其中的快慢冷却配置。
4. `burstErrorRetryMs` 作为批量模式统一错误重试等待时间，默认沿用现有 `60s` 语义。

## 9. 调试页可观测性

调试页至少需要展示以下信息：

1. `manual` 当前任务
2. `manual-burst` 当前任务与队列长度
3. `auto-burst` 当前任务与队列长度
4. Burst 共享状态机：
   - 当前阶段：`fast | slow | cooldown`
   - 当前阶段已消耗任务数
   - 当前阶段剩余预算
   - 下一次允许执行时间
5. 批量模式当前阻塞任务：
   - 通道名
   - 任务标识
   - 最近一次错误信息
   - 下次重试时间

目标是让用户能直接看出：

- 当前到底是被预算冷却拦住；
- 还是被某个失败任务阻塞；
- 以及阻塞发生在哪个通道。

## 10. 与现有实现的主要差异

相较当前实现，本方案的关键变化是：

1. 现有单一 `burst` 队列将被拆为 `manual`、`manual-burst`、`auto-burst` 三种逻辑通道。
2. `REQUEST_AUTHOR_PAGE` 与作者级刷新按钮不再进入批量 Burst 预算体系。
3. `manual-burst` 与 `auto-burst` 共享同一个 Burst 快慢冷却状态，而不是各自独立计数。
4. 常规更新不共享 Burst 预算状态，但仍继续在 Burst 类通道空闲后执行。
5. 调试页需要新增对 Burst 阶段状态与阻塞错误的可视化。

## 11. 实现边界

本方案当前只定义调度语义，不强行规定具体代码拆分方式。

实现时至少需要满足：

1. 现有作者任务去重规则仍然成立。
2. 现有请求批次与排序元数据规则仍然成立。
3. `manual-burst` 与 `auto-burst` 即使分成两个队列，也必须共享同一个 Burst 运行时状态对象。
4. `manual` 的失败通知语义不能退化成批量模式。
