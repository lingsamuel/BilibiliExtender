# 分组动态扩展 Spec（Chrome/Edge + Firefox 本地调试 + Bilibili）

## 1. 背景与目标

### 1.1 背景
Bilibili 现有关注/动态入口信息密集，无法按用户自定义主题分组查看投稿更新。

### 1.2 目标
实现一个 Chrome/Edge 扩展，在用户登录态下请求 Bilibili 接口，允许用户将“自己的收藏夹”定义为分组，并在站点 Header 注入“分组动态”入口，按分组展示收藏夹内作者的投稿视频更新。

### 1.3 非目标
- 不实现桌面通知。
- 不实现跨设备强一致同步（仅依赖浏览器 `storage.sync` 能力，受配额限制）。
- 不支持手动输入收藏夹 ID（首版必须从“我的收藏夹列表”选择）。

## 2. 术语
- 分组：由一个“我的收藏夹”映射而来（1:1）。
- 全部分组：虚拟聚合分组，聚合所有启用中的真实分组，不直接绑定收藏夹；可提供临时展示过滤，但不拥有独立的分组级近期基线。
- 时间流模式：围绕“近期投稿”目标的混排视图，按发布时间倒序；支持临时切到“全部”查看，但该状态不写回持久化。
- 近期投稿模式：按作者分段展示近期投稿，和时间流共享同一组“近期”基线。
- 全部投稿模式：按作者浏览全部投稿，仅影响展示，不参与近期追踪基线与 unread 计算。
- 作者基础顺序：按作者视图在“未勾选按更新时间倒序”时使用的顺序。若当前分组存在手动顺序记录，则按“手动顺序中的当前作者 + 自动顺序中新作者尾追加”合成；否则回退到自动顺序。
- 内建近期选项：`7天内 / 14天内 / 30天内` 三个固定选项。
- 默认近期范围：设置页定义的 `1-30` 天整数，默认 `7天内`；若不是 `7 / 14 / 30`，则在下拉中额外显示该自定义选项。
- 分组“上次看到”时间点：用户通过“上次看到这里”手动写入的自定义时间点；它与 `N天前` 一样，都是分组近期基线的一种取值。
- 有效已阅基线：某作者在某分组下用于红点和“近期投稿”判定的最终基线。优先取作者级已阅时间点，否则取当前分组近期基线（可能是 `N天前`，也可能是具体日期）。
- 时间语义统一规则：
  - 所有 `N天内` 都按整天分桶计算，不使用严格 `N * 24 小时` 窗口；
  - 实现上等价于：`N天内` 会覆盖到 `N天23小时59分钟` 这一档；
  - 投稿相对时间统一显示为 `X分钟内 / X小时前 / X天前`，避免出现 `0天前`。
- 全部投稿筛选：仅影响“全部投稿”当前展示、不参与 unread 计算的筛选（如 全部/7/14/30 天、10/30 条）。

## 3. 用户故事
1. 作为登录用户，我可以从“我的收藏夹列表”里选择若干收藏夹，创建分组并设置分组别名。
2. 作为用户，我在 `www.bilibili.com` 与 `space.bilibili.com` 顶部 Header 都能看到“分组动态”入口。
3. 作为用户，我点击入口后看到右侧抽屉面板，并能在左侧分组栏切换不同分组。
4. 作为用户，我可以切换“时间流/近期投稿/全部投稿”三种视图查看投稿。
5. 作为用户，我可以分别手动触发“刷新投稿列表”与“刷新收藏夹”；若距离上次刷新超过阈值（默认 10 分钟），系统自动刷新。
6. 作为用户，我在未查看某分组且出现新投稿时，该入口与对应分组显示红点；红点始终基于有效已阅基线计算，不因打开分组自动清除，也不受当前临时展示模式影响。
7. 作为用户，我在“近期投稿/全部投稿”模式可勾选“按更新时间倒序”；未勾选时，我可以拖动侧边栏作者列表重排，并记住该分组上次的手动顺序与勾选状态。
8. 作为用户，我可以使用默认“全部”分组聚合浏览所有分组，并通过临时“近期范围”快速收窄展示结果，而不改写任何真实分组的追踪状态。
9. 作为用户，我可以对分组或作者设置“忽略 unread 计数”，避免特定来源影响红点。

## 4. 功能需求

### 4.1 分组管理
- 数据来源：仅“我创建的收藏夹”。
- 创建分组：从收藏夹下拉/列表选择一个收藏夹后创建。
- 约束：一个收藏夹最多绑定一个分组（1:1）。
- 分组字段：
  - `groupId`（扩展内部 ID）
  - `mediaId`（收藏夹 ID）
  - `mediaTitle`（收藏夹原名）
  - `alias`（可选别名）
  - `enabled`（启用状态）
  - `excludeFromUnreadCount`（是否忽略该分组 unread 计数，默认 `false`）
  - `createdAt` / `updatedAt`
- 支持编辑：别名、启用状态、是否忽略 unread 计数。
- 支持删除分组。

#### 4.1.1 默认“全部”分组

- 系统提供虚拟聚合分组“全部”，默认开启（可在设置中关闭）。
- “全部”分组不绑定收藏夹，不写入 `GroupConfig`，也不保存独立的 `GroupReadMark` 历史。
- “全部”分组聚合范围：所有 `enabled === true` 的真实分组。
- “全部”分组的自动作者顺序：按启用分组在左侧分组栏中的顺序依次拼接各自 `authorMids`，再按首次出现去重。
- “全部”分组仅提供聚合浏览能力：
  - 允许在“时间流/近期投稿”中使用临时 `近期范围` 过滤当前展示；
  - 该过滤不持久化、不写入 `savedRecentDays` / `savedReadMarkTs`，离开分组后恢复设置页默认近期范围；
  - 该过滤不生成“上次看到”时间点，不显示也不改写分组级已阅能力。
- “全部”分组允许在按作者视图下保存独立的手动作者顺序：
  - 默认仍按自动聚合顺序展示；
  - 一旦用户发生拖拽重排，后续未勾选“按更新时间倒序”时优先使用该手动顺序；
  - 新增作者不插回理论聚合位置，而是按当前自动聚合顺序追加到末尾；
  - 暂时消失的作者不立即从手动顺序记录中删除，仅在当前渲染时过滤；若后续重新出现，应恢复到原先手动顺序中的相对位置。

### 4.2 入口注入与面板
- 注入页面：
  - `https://www.bilibili.com/*`
  - `https://space.bilibili.com/*`
- 注入位置：站点顶部 Header 区域，新增“分组动态”入口（视觉风格贴近站点现有导航）。
- 点击行为：打开右侧抽屉面板（内容脚本挂载 Vue 应用）。
- 面板结构：
  - 左侧：分组列表（支持红点）
  - 右侧：当前分组内容区 + 视图模式切换 + 刷新按钮 + 刷新状态（上次刷新时间 / 正在更新中）

#### 4.2.1 Header 入口红点/数字展示

- Header 入口支持两种状态：
  - 红点状态：`unreadCount > 0` 时显示。
  - 数字状态：显示全局 `unreadCount`，`>99` 时显示 `99+`。
- 红点/数字元素复用站点样式类 `red-num--message`，避免自定义视觉与站点风格偏差。
- `unreadCount = 0` 时数字文本为空（隐藏数字），不通过 `display` 切换元素显隐。
- 红点/数字挂载的父节点必须显式设置 `position: relative`，保证绝对定位稳定，不受站点布局变更影响。

#### 4.2.2 抽屉导航状态模型

抽屉侧栏统一使用“导航条目 ID（`activeEntryId`）”驱动，不再使用独立的 `showSettings` / `showDebug` 布尔开关。

- 真实分组条目：使用分组自身 `groupId`。
- 虚拟分组条目（参与 feed 展示）：
  - 全部分组：`__bbe_all__`
- 页面虚拟条目（仅用于页面切换）：
  - 设置页：`__bbe_settings__`
  - 调试页：`__bbe_debug__`

交互规则：
- 点击“全部”后，`activeEntryId = __bbe_all__`，展示聚合后的 feed。
- 点击“设置”后，`activeEntryId = __bbe_settings__`，显示设置页。
- 点击“调试”后，`activeEntryId = __bbe_debug__`，显示调试页。
- 再次点击“设置”或“调试”时，不自动切回分组页（无 toggle 返回行为）。
- 点击任一真实分组后，`activeEntryId = groupId`，显示对应分组内容。

实现约束：
- 当 `activeEntryId` 为真实分组 ID 或 `__bbe_all__` 时，允许触发 feed 请求。
- 设置/调试条目不参与分组缓存、未读统计与分组持久化字段（如 `lastGroupId`）。
- “全部”分组不写入 `GroupConfig`，但可参与聚合 unread 展示与临时展示过滤。

### 4.3 投稿聚合规则
- 输入：分组绑定收藏夹中的视频集合。
- 作者提取：从收藏夹视频中提取作者 `mid` 去重。
- 投稿拉取：按作者获取投稿视频列表（仅用于视频列表数据）。
- 作者信息拉取：统一使用 `x/web-interface/card` 获取作者信息（`name`、`face`、`follower`、`following`）。
- 去重：按视频 `bvid` 去重。
- 排序：按投稿发布时间（`pubdate`）倒序。

#### 4.3.1 作者信息来源约束
- 作者名称、头像、粉丝数、关注状态的展示与业务判断，均以 Card API 返回结果为准。
- 仅在“该作者没有任何已缓存 Card 信息”且本轮 Card 请求失败时，允许临时回退使用投稿接口中的作者名兜底展示。
- 一旦该作者已存在 Card 缓存，后续即使 Card 请求失败也必须继续使用缓存，不回退到投稿接口作者信息。

### 4.4 展示模式

- 工具栏模式按钮为：`时间流 / 近期投稿 / 全部投稿`。
- 三种模式的职责划分：
  - `时间流`：按时间顺序查看近期投稿，支持临时切到 `全部` 查看。
  - `近期投稿`：按作者查看近期投稿，不再提供 `全部` 筛选。
  - `全部投稿`：按作者查看全部投稿，仅影响展示，不参与近期追踪。
- “近期”的判定统一来自同一套分组级设置：
  - 设置页定义的 `N天前`，允许 `1-30` 任意整数，默认 `7天前`；
  - 自定义时间点：分组“上次看到”时间点；
  - 两者地位同级，都会作为分组持久化的近期基线参与 unread。
- 作者级已阅时间点优先级高于分组级设置；该优先级在三种模式下保持一致。

#### 4.4.1 时间流模式
- 模式目标：按时间顺序查看“近期投稿”。
- 时间流与“近期投稿”共享同一组分组级近期设置：
  - 一个持久化的分组近期基线；
  - 基线可取设置页定义的 `N天前`，也可取某个具体时间点。
- 时间流额外提供一个临时展示选项：`全部`。
  - `全部` 只影响当前时间流列表展示；
  - `全部` 不写入分组持久化状态；
  - `全部` 不改变分组红点，不改变 Header unread，也不改变“近期投稿”模式的结果。
- 下拉触发器文案规则：
  - 使用 `N天前` 基线时，显示 `N天前`；
  - 存在分组“上次看到”时间点时，仅显示最新一个时间点，不展示完整历史列表。
- 下拉选项规则：
  - 时间流始终显示临时 `全部`；
  - 时间流与近期投稿始终显示当前设置页定义的 `N天前`；
  - 若存在分组“上次看到”时间点，再额外显示最新一个具体时间点；
  - 用户可在 `N天前` 和具体时间点之间来回切换，两者都会写入分组持久化基线。
- 默认初始目标数量：`50`（可配置项）。
- 滚动增量目标数量：每次触底追加 `20`（固定值，首版不暴露配置）。
- “加载更多”按钮与触底自动加载并存：
  - 当 `hasMoreForMixed = true` 时，列表底部显示“加载更多”按钮；
  - 当 `hasMoreForMixed = false` 时，隐藏“加载更多”按钮，仅显示“没有更多内容了”；
  - 保留触底自动加载行为，两者复用同一 loadMore 链路与同一节流状态。
- 重要规则：
  - 为构造目标数量，可对作者列表执行多轮拉取；
  - 若某轮请求后数量超过目标值，**不裁剪**，全部展示；
  - 仅在“当前可用数据小于目标值”时继续拉取下一轮。

#### 4.4.1.1 时间流按天分段与左侧时间轴
- 时间流列表按“有投稿的自然日”分段，不显示分段标题文案，仅在分段之间保留视觉留白。
- 左侧时间轴与内容滚动联动，按“有投稿日期”渲染节点。
- 时间文案规则：
  - 当天显示 `今天`；
  - 昨天显示 `昨天`；
  - 距离今天 2–7 天（含）显示 `N 天前`（如 `3 天前`）；
  - 其余日期显示 `MM-DD`。
- 节点样式状态：
  - 日期分段在可视区域内：显示空心圆节点；
  - 日期分段不在可视区域内：退化为小黑点节点。
- 时间线节点交互：
  - 仅真实分组支持时间线节点的分组级已阅交互；
  - 左键点击任一节点，可将当前真实分组的已阅基线设置到该日期分界；
  - 仅当某节点当前处于“已阅对勾激活态”时，才允许右键点击该节点清除当前真实分组的自定义已阅基线；
  - 右键清除后，当前真实分组立即恢复到设置页定义的默认近期范围基线；
  - 在“全部”分组中，时间线节点仅用于导航，不提供左键设基线或右键清基线。
- 时间轴压缩窗口规则：
  - 可视区域内的“有投稿日期”节点全部保留，不做压缩；
  - 在可视区域之外，仅额外保留上方 2 个与下方 2 个“有投稿日期”节点；
  - 离屏节点固定停靠在时间轴顶部/底部区域，不贴靠可视节点，避免滚动到临界位置时跳变；
  - 无投稿的日期不占位、不参与前后计数；
  - 不允许无限累积离屏日期节点。

#### 4.4.1.2 时间流“上次看到这里”分割
- 分割线支持任意已阅时间点，不再要求命中自然日 `00:00` 边界。
- 当当前分组近期基线为某个具体时间点时，该时间点成为时间流与“近期投稿”共享的有效基线。
- 在混排结果中定位第一条“`pubdate < readMarkTs`”的视频，并在该卡片前插入“上次看到这里”横向分割线。
- 若列表首条视频已满足 `pubdate < readMarkTs`，则分割线显示在列表顶部。
- 分割线仅作为视觉分隔，不触发“截断后方旧内容”。
- 即使时间流当前临时切换到 `全部`，该分割线仍按有效基线绘制，用于提示用户最近一次看到的位置。
- “全部”分组不显示该分割线，因为它没有独立的分组级已阅基线。

#### 4.4.1.3 时间流“撤销上次看到”交互
- 当分组存在“上次看到”时间点时，时间流工具栏显示 `撤销上次看到` 按钮。
- 点击后撤销最近一次由“上次看到这里”写入的分组级自定义时间点：
  - 若撤销后仍有更早的自定义时间点，则继续使用最新一条作为有效基线；
  - 若撤销后已无自定义时间点，则恢复当前持久化的 `N天前` 基线。
- 撤销结果会同时影响“时间流”和“近期投稿”两种模式。
- “全部”分组不显示 `撤销上次看到`，因为它不生成分组级“上次看到”时间点。

#### 4.4.1.4 时间流展示下界与平滑策略
- 时间流展示必须保留“至少当前设置页定义的 N 天窗口”的可见内容（默认 `7天内`）。
- 即使存在分组/作者级已阅时间点，也不按已阅时间硬截断旧内容。
- 展示下界取“最近 N 天窗口下界”与“当前数据最旧可用时间”的平滑组合，避免切换已阅点时出现突变式折叠。

#### 4.4.1.5 时间流“卡片间设置已阅”交互
- 在相邻 VideoCard 之间 hover 时，显示半透明粉色竖线虚影，仅作为交互提示。
- 点击竖线虚影后，将当前分组已阅时间点设置为“两条视频之间”的时间：
  - 常规取值：`olderTs + 1`；
  - 同秒不可切分时退化为 `newerTs`。
- 设置成功后，持久分割统一使用“上次看到这里”横向分割线，不保留竖线作为持久分隔。

#### 4.4.2 近期投稿模式
- 模式目标：按作者查看近期投稿。
- “近期投稿”不再提供 `全部` 筛选，只显示命中有效已阅基线之后的近期内容。
- 有效已阅基线规则：
  - 优先使用作者级已阅时间点；
  - 否则使用当前分组持久化的近期基线；
  - 该基线可为 `N天前`，也可为某个具体时间点。
- 总量不设上限（由作者数量决定）。
- 每位作者内部按发布时间倒序。
- 每位作者标题右侧展示关注按钮：
  - 未关注：`+ 关注 X万`；
  - 已关注：`已关注 X万`；
  - `X万` 基于 Card API 的 `follower` 字段格式化。
- 作者标题行布局分为左右两组：
  - 左侧操作组：头像 + 作者名链接 + 关注按钮；
  - 右侧操作组：作者级 `标记已阅 / 撤销已阅` + `不计算未读`。
- 点击按钮可直接关注/取消关注（`x/relation/modify`）：
  - 前端先做乐观更新；
  - 后台请求成功后回写最新 Card 信息；
  - 失败时回滚并以通知弹框提示，不替换主内容区。
- 关注请求结构要求：
  - 必须携带 `extend_content`（JSON 字符串：`{"entity":"user","entity_id":fid}`）；
  - 当前实现固定补齐 `gaia_source=web_main`、`spmid=333.1387`、`is_from_frontend_component=true`；
  - 当前实现固定补齐 URL Query：`statistics={"appId":100,"platform":5}` 与 `x-bili-device-req-json={"platform":"web","device":"pc","spmid":"333.1387"}`；
  - `extend_content` 允许经 `URLSearchParams` 编码后提交，不要求保留抓包中的原始冒号字面值；
  - 请求当前由 background 发起，但会在发起前通过 DNR session rule 把 `Origin`、`Referer`、`Sec-Fetch-Site` 改写为贴近页面前端的值。
- 作者级已阅操作规则：
  - `标记已阅` 按钮始终显示，点击后把当前时间戳（`now`）压入作者级已阅历史栈；
  - 当作者已经存在显式作者级已阅时间点时，额外显示 `撤销已阅`；
  - 点击 `撤销已阅` 后只回退最近一次作者级已阅写入；
  - 右键作者边界线仍保留“清空全部作者级已阅历史”的能力；
  - 作者级已阅历史栈跨分组共享，最多保留 10 条，因此撤销结果会同步影响该作者在所有分组中的近期判定与红点计算。
- “不计数”按钮文案统一改为“`不计算未读`”。
- 按作者视图（“近期投稿”与“全部投稿”）的工具栏新增勾选项：`按更新时间倒序`。
- 勾选项默认值：`true`（默认勾选）。
- 作者基础顺序定义：
  - 普通分组：默认取收藏夹作者原始顺序（`authorMids` 顺序）。
  - “全部”分组：默认取当前启用分组的自动聚合顺序。
  - 若当前分组存在手动顺序记录，则改为“手动顺序中的当前作者 + 自动顺序中新作者尾追加”。
- 勾选为 `true` 时：
  - 按“该作者在当前筛选条件下可见视频的最新 `pubdate`”倒序排列作者列表；
  - 若某作者在当前筛选条件下无可见视频，则排在列表末尾；
  - 若最新 `pubdate` 相同，按作者基础顺序稳定排序。
  - 排序完成后，若存在至少一位作者的最新 `pubdate` 严格早于当前有效基线，则在第一位满足该条件的作者前插入与时间流相同的“上次看到这里”横向分割线。
  - 这里的“当前有效基线”与时间流完全共用同一语义：既可能是当前持久化的 `N天前` 基线，也可能是用户通过时间流“上次看到这里”写入的具体时间点。
- 勾选为 `false` 时：作者列表按作者基础顺序展示。
- 勾选状态按分组维度持久化记忆，切换分组后恢复各自上次选择；同一分组的“近期投稿/全部投稿”共享这份勾选状态。
- 手动顺序按分组维度持久化记忆；同一分组的“近期投稿/全部投稿”共享同一份手动顺序。
- “全部”分组下仍可勾选 `按更新时间倒序`，但它只影响当前聚合浏览展示，不参与任何分组级追踪状态。
- 工具栏“已阅”按钮文案：
  - 普通分组：`全部标记已阅`（语义为将当前分组追踪基线推进到 `now`）；
  - “全部”分组：不显示分组级“已阅”按钮。

#### 4.4.2.1 按作者模式子侧栏导航
- 在“近期投稿”与“全部投稿”模式显示子侧栏，位置在内容区左侧。
- 子侧栏为竖向列表，不使用时间线样式，不增加额外分段间距。
- 列表项展示：作者头像 + 作者名字（不显示 mid 数字）。
- 点击列表项后，内容区平滑滚动到对应作者分组顶部。
- 子侧栏需要独立滚动；当作者数量较多时，不影响右侧内容区滚动。
- 滚动联动高亮：内容区滚动时，子侧栏自动高亮当前可视区域内的作者。

#### 4.4.2.1.1 按作者模式侧边栏拖拽重排
- 仅在“近期投稿/全部投稿”且未勾选 `按更新时间倒序` 时，子侧栏作者列表允许拖拽重排。
- 拖拽完成后，主内容区作者分段与子侧栏必须同步使用同一顺序，不能只重排导航不重排正文。
- 首次拖拽时，若当前分组尚无手动顺序记录，则以当前作者基础顺序作为初始快照，再应用本次拖拽结果。
- 手动顺序持久化规则：
  - 按分组维度保存；
  - “近期投稿”和“全部投稿”共用同一份；
  - “全部”分组拥有自己的独立手动顺序，不与任何真实分组合并。
- 分组刷新、收藏夹作者列表变化、启用分组变化后，当前作者基础顺序的合成规则为：
  - 先取手动顺序中当前仍存在的作者，保持其相对顺序不变；
  - 再把当前自动顺序里“尚未出现在手动顺序中的新作者”按自动顺序追加到末尾。
- 对“全部”分组，上述“自动顺序”指当前启用分组重新聚合后的顺序；一旦存在手动顺序，后续不再整体回退到纯自动聚合，只做“保留既有手动顺序 + 新作者尾追加”。
- 若某作者只是暂时不在当前作者集合中，不立即从手动顺序记录中物理删除；当前渲染时仅跳过该作者，若未来重新出现，应恢复其原手动位置。

#### 4.4.2.1.2 近期投稿“上次看到这里”分割
- 仅在“近期投稿”模式下勾选 `按更新时间倒序` 时显示该分割线。
- “全部”分组不显示该分割线，因为它没有分组级已阅基线可供比较。
- 分割线样式与文案完全复用时间流中的“上次看到这里”横向分割线，不额外引入新文案。
- 插入规则：
  - 先按当前排序后的作者列表，自上而下计算每位作者的“最新投稿时间”；
  - 定位第一位“最新投稿时间严格早于当前有效基线”的作者；
  - 在该作者分段前插入“上次看到这里”横向分割线；
  - 若排序后的首位作者已经满足该条件，则分割线显示在列表顶部。
- 这里比较使用的“最新投稿时间”基于作者全量缓存中的最新 `pubdate`，而不是仅看当前作者段里实际展示的卡片。
- 若所有作者的最新投稿时间都不早于当前有效基线，则不显示分割线。

#### 4.4.2.2 作者“一键点赞”
- 在作者标题左侧操作组（关注按钮同级）新增“一键点赞”按钮。
- 点击后对该作者“当前界面可见视频卡片”执行批量点赞（仅当前筛选 + 当前分页，不隐式拉取未展示页）。
- 批量点赞通过通道化调度器中的 `like-action` 通道串行执行，避免并发请求触发风控。
- 批量点赞成功的视频需把 `bvid -> likedAt` 写入 `chrome.storage.local`，仅保留 30 天，不写入 `storage.sync`。
- 按钮执行中显示 loading 并禁用；批次结束后以通知弹框汇总成功/失败数量。

#### 4.4.3 全部投稿模式
- 模式目标：按作者查看全部投稿，不承担“查看近期”的追踪语义。
- 全部投稿模式也显示 `按更新时间倒序` 勾选项，和“近期投稿”共用同一份按分组记忆的排序偏好。
- 全部投稿模式使用独立下拉（全部投稿组）：
  - `全部`
  - `7天内`
  - `14天内`
  - `30天内`
  - `10条`
  - `30条`
- 全部投稿模式不显示分组级“标记已阅”按钮。
- 过滤选项语义：
  - 只影响当前列表展示；
  - 不参与分组 unread 与 Header unread 计算；
  - 不写入当前分组的近期基线（`savedReadMarkTs` / `savedRecentDays`）。
- 全部投稿模式沿用按作者列表布局与作者级操作：
  - 保留关注按钮、作者级 `标记已阅 / 撤销已阅`、`不计算未读` 与“一键点赞”按钮；
  - 作者级已阅时间点在该模式下仍然跨分组统一，且可在此模式下清除。
- 全部投稿模式下若作者存在作者级已阅时间点：
  - 仅绘制作者级已阅标记线；
  - 不触发“已阅后只保留额外 N 条”的截断逻辑。
- `全部投稿 = 全部` 时：
  - 使用作者全量缓存参与展示；
  - 启用作者内分页；
  - 不受分组已阅时间点与作者级已阅时间点过滤影响。
- `全部投稿 = 7天内 / 14天内 / 30天内` 时：
  - 先按对应时间窗口筛选作者视频；
  - 若作者在窗口内命中至少 1 条视频，则只展示命中结果；
  - 若作者在窗口内 0 命中，但缓存中仍存在更旧视频，则补 1 条该作者最新视频作为兜底展示；
  - 兜底展示时在作者标题旁显示“最近更新：X分钟内 / X小时前 / X天前”提示；
  - 该兜底视频仅用于全部投稿展示，不改变 unread 计算，也不复用“已阅后额外 N 条”的逻辑。
- `全部投稿 = 10条 / 30条` 时：
  - 仅按条数截断当前作者列表；
  - 不做“保底补 1 条”。
- 全部投稿模式沿用同一套按作者子侧栏与拖拽重排规则；当未勾选 `按更新时间倒序` 时，作者列表顺序与“近期投稿”共享同一份分组级手动顺序。
- 只要处于时间流视图，若视频命中“作者级逻辑已阅”（`video.pubdate <= authorReadMarkTs`）：
  - VideoCard 默认半透明；
  - hover 恢复不透明；
  - 不透明度变化带 transition 动画。
- 细节行为以 `docs/read-mark-spec.md` 为准。

#### 4.4.3.1 作者内分页（仅“全部投稿 = 全部”）
- 当且仅当处于“全部投稿”模式，且筛选为 `全部` 时启用作者内分页。
- 当处于 `7天内 / 14天内 / 30天内 / 10条 / 30条` 时，不显示分页控件，沿用当前筛选结果直出。
- 分页容器（页码区 + 跳页区）在作者分段底部整体水平居中显示。
- 作者分页读取口径必须是“精确页结果”，不能再用共享缓存按旧 `sourcePn` 切片近似替代。
- 分页大小使用投稿 API 的 `ps` 参数值；若当前作者尚无接口返回值，则回退到高级设置 `authorVideosPageSize` 作为请求默认值。
- 每位作者分页状态独立维护，默认进入第 1 页。
- 分页控件能力：
  - `上一页/下一页`；
  - 页码按钮（页数较多时使用省略号）；
  - 跳页输入（支持直接跳到指定页）。
- 当目标页尚未缓存时：
  - 前台仅提交 `REQUEST_AUTHOR_PAGE` 意图（`groupId + mid + pn + ps`），不等待补页任务完成；
  - 后台按 `(mid, pn, ps)` 获取精确页结果，并把本次结果写入作者投稿块缓存，同时立即尝试重建作者连续缓存；
  - 调度器异步执行该请求；该通知能力仅服务于作者分页，不扩展到“刷新投稿列表”“刷新收藏夹”或时间流补页；
  - 若后台完成目标页获取，会主动向当前内容页发送“页已就绪”通知；前台收到且确认仍命中当前 pending 目标后，立即重新读取一次当前作者分页结果并自动切换到目标页；
  - 若后台执行目标页任务失败，会主动向当前内容页发送“页失败”通知；前台收到后立即停止该作者本次等待中的兜底轮询、清除 pending/loading 状态，并弹出错误提示；
  - 作为通知丢失时的兜底，作者分页仍保留快速轮询：前三次间隔 `500ms`，后续间隔 `1000ms`；一旦收到成功通知并完成刷新，或收到失败通知，即停止本次轮询；
  - 若作者已无更多页，返回 `no-more` 并提示用户当前最大可用页。
- 分组切换、全部投稿筛选切换后，作者分页重置为第 1 页。

- 作者级已阅标记线必须位于“前后两个视频卡片之间”的几何中线。
- 不允许贴靠更新较旧或较新的任一卡片边缘。

### 4.5 缓存与刷新机制

#### 4.5.1 作者投稿列表缓存（AuthorVideoCache）

作者投稿缓存不再以“页码是否已缓存”为核心真相，而是拆成“连续缓存 + 分块缓存”两层，按作者（`mid`）跨分组共享。

全局数据结构 `AuthorVideoCache`，按 `mid` 索引：

```ts
interface AuthorVideoVersionFingerprint {
  totalCount: number;
  tagCounts: Array<{ tid: number; count: number }>; // 按 tid 升序序列化
}

interface AuthorVideoBlock {
  pageNum: number;
  pageSize: number;
  startIndex: number; // (pageNum - 1) * pageSize
  endExclusive: number; // startIndex + videos.length
  fetchedAt: number;
  version: AuthorVideoVersionFingerprint;
  videos: VideoItem[]; // 保持该块内部的 API 返回顺序（按时间倒序）
}

interface AuthorVideoCache {
  mid: number;
  // 作者基础信息统一来自 Card API
  name: string;
  face?: string;
  follower?: number;
  following?: boolean;
  // 当前已经验证“从头开始无断层”的连续数组
  continuousVideos: VideoItem[];
  continuousVersion?: AuthorVideoVersionFingerprint;
  continuousUpdatedAt?: number;
  // 所有候选块：包含当前版本未接上的块、旧版本块、手动翻页/跳页块
  blocks: AuthorVideoBlock[];
  latestKnownVersion?: AuthorVideoVersionFingerprint;
  latestKnownTotalCount?: number;
  lastFirstPageFetchedAt?: number;
  lastFetchedAt: number;
}
```

存储位置：`chrome.storage.local`（key: `bbe:author-video-cache`）。

作者投稿列表服务（AuthorPostListService）统一负责以下能力：
- 刷新首页并提取版本摘要；
- 获取精确页结果（`pn + ps`）；
- 维护连续缓存；
- 维护块缓存；
- 处理时间流按需补块；
- 处理 `pageSize` 变化后“第一页/第二页不足”的补足逻辑。

版本摘要（Version Fingerprint）定义：
- 使用 search API 返回值中的 `data.page.count`；
- 使用 `data.list.tlist` 中各 tag 的 `(tid, count)` 集合；
- 若两次请求的上述摘要一致，则视为同一个作者投稿列表版本。

块缓存规则：
- 每次作者投稿请求都必须落成一个 `AuthorVideoBlock`，记录 `pageNum/pageSize/startIndex/endExclusive/version/videos`。
- 块缓存不按 `pageSize` 隔离；不同 `pageSize` 下的块统一按绝对序号区间理解。
- 若两个块版本一致，也不直接因为区间相邻就拼接；服务必须优先尝试以下接续证据：
  - 相同 `bvid`；
  - 时间戳交叉；
  - 仅当两者都不满足时，才把“绝对序号区间无缝衔接/重叠”视为最后的弱证据参与拼接。
- 手动翻页、跳页、时间流按需补块、后台预取得到的数据都先写入块缓存，再尝试重建连续缓存。

连续缓存规则：
- 连续缓存的真相是“当前已经验证可以从头无断层读取的一段数组”，而不是“缓存到了第几页”。
- 同版本下，服务应基于块缓存重建“从 `startIndex = 0` 开始连续覆盖的最大前缀”，但块与块之间的拼接顺序仍遵循“相同 `bvid` 优先、时间戳交叉次之、区间连续性最后兜底”的证据优先级，并写入 `continuousVideos`。
- 跨版本时，若新版本头部块与旧连续缓存之间能通过“相同 `bvid` 优先、时间戳接续兜底”证明可接续，则允许直接生成新的连续数组：
  - 新头部使用新版本块；
  - 旧连续尾部只要能正确接到新头部后方，就可吸收到新的连续数组中；
  - 接续完成后，连续缓存整体提升为新版本，不需要为旧数组重新计算理论坐标，只以新数组实际长度作为连续覆盖长度。
- 即使连续缓存已经构造完成，后续刷新仍必须允许向中间插入新增数据或修补此前暂不可见的视频；连续缓存不是冻结快照。

容量与淘汰：
- 连续缓存固定优先保留“2 个 `pageSize` 的浅层窗口”，用于稳定支撑第一页和第二页展示。
- 在此基础上，再额外保留 `authorContinuousExtraPageCount * authorVideosPageSize` 条连续数据：
  - `authorContinuousExtraPageCount` 默认 `3`，最小 `1`。
- 因此连续缓存总目标长度为 `(2 + authorContinuousExtraPageCount) * authorVideosPageSize`。
- 块缓存单独限额，最多保留 `authorNonContinuousCachePageCount * authorVideosPageSize` 条等价容量：
  - `authorNonContinuousCachePageCount` 默认 `3`，最小 `1`。
- 连续缓存与块缓存都按“最旧优先”淘汰；旧版本块不立即丢弃，允许保留到超出容量后再清理。

刷新行为：
- 默认刷新第一页（`pn=1`，`ps=authorVideosPageSize`）：
  - 过期时优先刷新第一页，更新 `lastFirstPageFetchedAt`，并立即重建连续缓存；
  - 首页未过期时，不重复拉首页。
- 连续窗口维护：
  - 调度器应尽量维护“连续缓存总目标长度”；
  - 第二页仍是默认优先的防御性目标，但不再要求“页级 usedInMixed 回报”才能预取；
  - 若当前连续长度不足以覆盖第一页、第二页或配置要求的额外连续窗口，则可继续按需抓取后续页块并重建连续缓存。
- `pageSize` 变化补足：
  - 若用户修改 `authorVideosPageSize` 后，当前连续缓存长度不足以支撑新的第一页或第二页，则应复用同一套“按需补块 + 重建连续缓存”逻辑补足，而不是简单把旧页号视为仍然准确。
- “刷新投稿列表”与作者后台常规首页刷新允许同步请求一次 Card API，更新 `name/face/follower/following`。
- “刷新收藏夹”只更新分组标题与作者列表，不得触发任何作者投稿请求或 Card API 请求。
- 若本轮 Card 请求失败：
  - 有已有 Card 缓存：保留并继续使用已有缓存；
  - 无 Card 缓存：允许临时使用投稿接口作者名兜底，直到后续 Card 刷新成功。
- “刷新投稿列表”确认后等价于 `forceRefresh`：无视过期判定，强制重新拉取首页并重建连续缓存。

#### 4.5.2 分组与作者的关系

分组不再直接持有视频数据，改为持有作者 mid 列表的引用：

```ts
interface GroupFeedCache {
  groupId: string;
  authorMids: number[];       // 该分组包含的作者 mid 列表
  updatedAt: number;          // 作者列表最后更新时间（从收藏夹重建的时间）
}
```

分组刷新流程：
1. 调用 `getAllFavVideos(mediaId)` 获取收藏夹内容，提取作者列表 → 更新 `GroupFeedCache.authorMids`。
2. 对列表中每个 `mid`，检查 `AuthorVideoCache[mid]` 是否过期，过期则刷新。
3. 组装分组视图时，从全局 `AuthorVideoCache` 中按 `authorMids` 聚合数据。

跨分组共享效果：分组 A 刷新了 UP 主 X，分组 B 也包含 UP 主 X 时直接命中缓存。

#### 4.5.3 收藏夹列表缓存

`getAllFavVideos(mediaId)` 的结果也纳入缓存，按 `mediaId` 索引，过期时间同样遵守 `refreshIntervalMinutes`。避免同一分组短时间内重复拉取收藏夹全量数据。

#### 4.5.4 其他请求缓存

| 请求 | 缓存粒度 | 缓存位置 | 过期时间 |
|------|----------|----------|----------|
| `getMyCreatedFolders` | 全局单例 | 内存 | `refreshIntervalMinutes` |
| `getCurrentUser` | 全局单例 | 内存 | `refreshIntervalMinutes` |
| `getWbiKeys` | 全局单例 | 内存 | 2 分钟（固定），遇 412 自动清除重试 |

#### 4.5.5 前台刷新（用户触发）

- 手动刷新入口拆分为两类，抽屉工具栏与设置页分组行都保持一致：
  1. “刷新投稿列表”：
     - 点击后必须先进行二次确认；
     - 确认后立即向调度器优先入列该分组的收藏夹刷新任务（`group-fav`）；
     - `group-fav` 成功后，基于最新作者列表按缓存状态衔接作者任务：无缓存入 Burst；有缓存且过期入 `author-video` 常规优先；未过期不入队；
     - 前台通过轮询 `GET_GROUP_FEED` 获取最新缓存，不等待刷新链路同步完成。
  2. “刷新收藏夹”：
     - 点击后不需要二次确认；
     - 只向调度器优先入列该分组的 `group-fav` 任务；
     - 任务完成后仅更新分组标题与作者列表缓存，不继续衔接任何作者投稿刷新任务。
- 新增分组（`UPSERT_GROUP` 新建）：
  1. 保存成功后立即向调度器优先入列该分组的 `group-fav` 刷新任务。
  2. 编辑已有分组不自动触发该刷新（避免重复请求）。
- 自动刷新（打开面板或切换分组）：
  1. 若分组无 `GroupFeedCache`：提交 `group-fav` 优先任务，返回 `cacheStatus: 'generating'`。
  2. 若分组已有 `GroupFeedCache`：不自动触发 `group-fav`（收藏夹缓存仅由定时调度、“刷新投稿列表”与“刷新收藏夹”驱动）。
  3. 若分组作者缓存仅部分命中：提交缺失作者的 `author-video` 优先任务，返回 `cacheStatus: 'generating'`，并携带当前可展示的缓存结果（不清空已有内容）。
  4. 仅当“本轮需要补齐的作者都完成至少一轮缓存（存在 `AuthorVideoCache[mid].lastFetchedAt`）”后，才返回 `cacheStatus: 'ready'`。
  5. 若当前完全没有可展示内容，则显示“正在生成缓存”；若已有部分内容，则保持内容可见并显示“正在更新中”状态。
- 前台调度边界：
  1. 前台只允许“读缓存 + 提交刷新意图”；可提交 `REQUEST_AUTHOR_PAGE(groupId, mid, pn, ps)` 表达目标页需求，但不能指定调度细节（如队列位置、是否 `failFast`）。
  2. 前台请求不得同步等待调度任务完成；作者分页场景通过“后台完成通知 + 立即补拉一次分页数据读取 + 快速兜底轮询”感知结果，其他刷新链路仍通过后续轮询/下一次读取自然可见。

#### 4.5.6 刷新状态展示

- 工具栏右侧状态位规则：
  - `ready`：显示相对时间（如“刚刚刷新 / X 分钟前”）。
  - `generating` 且无可展示内容：显示“正在生成缓存，请稍候...”。
  - `generating` 且已有可展示内容：显示转圈动画 + 文本“正在更新中”，不显示“上次刷新时间”。

#### 4.5.7 后台定时刷新

通过通用调度器 + `chrome.alarms` 实现，Service Worker 休眠后仍可被唤醒。调度细节见 `docs/scheduler-spec.md`。

后台刷新拆分为两条独立通道（互不共享 ratelimit）：
1. `author-video`：刷新作者视频缓存，周期由 `backgroundRefreshIntervalMinutes` 控制。
2. `group-fav`：刷新收藏夹缓存（标题 + 作者列表），周期由 `groupFavRefreshIntervalMinutes` 控制。

批次执行规则保持不变：每个通道都按 `schedulerBatchSize` 分批、批内串行 + 固定间隔、批间按周期均匀分散，并保留失败任务重试机制；详见 `docs/scheduler-spec.md` 3.5 节。
当存在“作者无缓存”目标时触发 Burst 模式：Burst 优先于常规调度，常规任务会在 Burst 清空后恢复；详见 `docs/scheduler-spec.md` 3.6 节。

#### 4.5.8 时间流分页构造与补页

时间流在构造目标片段（如 `1-50`、`51-70`、`71-90`）时遵循以下规则：

1. 时间流优先使用每位作者当前的 `continuousVideos` 构造结果；块缓存只作为补块后的连续重建输入，不直接充当时间流展示真相。
2. 若构造过程中发现某作者当前连续缓存长度不足，且仍可能贡献当前筛选可见数据，则仅向调度器提交“边界补块意图”；本次读取不等待任务完成。
3. 边界补块由作者投稿列表服务决定具体请求的 `pn + ps`；默认优先使用当前 `authorVideosPageSize` 口径，并先尝试补足第二页，再扩展更深的连续窗口。
4. 每次补块成功后，必须先写入块缓存，再立即尝试重建连续缓存；时间流下一次读取拿到的应是重建后的连续结果。
5. 若 Burst 补块失败：记录调度历史并在状态面板可观测，不中止当前缓存读取链路。
6. 时间流补块、`pageSize` 变化补足、手动翻页回灌三者共享同一套“写块 -> 重建连续缓存”机制，不再依赖旧的页级 `usedInMixed` 推进链。

#### 4.5.9 设置项汇总

| 设置项 | UI 标签 | 含义 | 默认值 | 范围 |
|--------|---------|------|--------|------|
| `refreshIntervalMinutes` | 请求缓存时长（分钟） | API 请求结果的缓存有效期 | 30 | 1–120 |
| `backgroundRefreshIntervalMinutes` | 作者缓存刷新间隔（分钟） | `author-video` 通道 alarm 周期 | 10 | 5–120 |
| `groupFavRefreshIntervalMinutes` | 收藏夹缓存刷新间隔（分钟） | `group-fav` 通道 alarm 周期 | 10 | 5–120 |
| `schedulerBatchSize` | 每批任务数 | 调度器每个通道每批最多执行任务数（全通道共享） | 10 | 1–50 |
| `authorVideosPageSize` | 作者投稿每页数量 | 作者投稿接口 `ps` 参数默认值，并作为分页展示兜底口径 | 24 | 1–42 |
| `authorContinuousExtraPageCount` | 作者连续缓存额外页数 | 在固定保留 2 页之外，额外保留的连续缓存页数 | 3 | 1–10 |
| `authorNonContinuousCachePageCount` | 作者非连续缓存页数 | 分块候选缓存的容量上限（按默认 `pageSize` 折算） | 3 | 1–10 |
| `enableAllGroup` | 显示“全部”分组 | 是否在侧栏显示默认聚合分组 | `true` | `true/false` |

### 4.6 红点（未读）规则
- 分组红点与“追踪基线”绑定，不使用“打开分组即已读”的模型。
- 分组级未读计算：
  - `excludeFromUnreadCount === true` 的分组，未读数固定为 `0`；
  - 全部投稿筛选（全部/7/14/30 天、10/30 条）不参与未读计算；
  - 作者若命中“忽略 unread 计数”开关（跨组共享），其视频在所有分组中均不计入 unread；
  - 作者若存在作者级已阅时间点，则该作者使用作者级基线（绝对覆盖分组基线，无关大小）；
  - 其余作者按分组有效已阅基线计算：直接使用当前分组持久化的近期基线（可能是 `N天前`，也可能是某个具体时间点）；
  - 时间流中的临时 `全部` 只影响当前展示，不参与上述基线选择；
  - 分组未读数 = 分组内去重后视频中同时满足：
    - `pubdate > baselineTs`
    - 且视频最终状态为“未已阅”。
- Header 全局 unread count 采用“按作者聚合”规则：
  - 聚合时忽略被分组开关或作者开关排除的来源；
  - 同一作者跨分组仅按 1 份视频集合去重计数；
  - “全部”分组是聚合视图，不额外放大或改变其他分组基线。

### 4.7 设置项
- `refreshIntervalMinutes`：请求缓存时长（默认 30）。
- `backgroundRefreshIntervalMinutes`：作者缓存刷新间隔（默认 10）。
- `groupFavRefreshIntervalMinutes`：收藏夹缓存刷新间隔（默认 10）。
- `schedulerBatchSize`：调度器每批任务数（默认 10，全通道共享）。
- `timelineMixedMaxCount`：时间流模式目标基数（默认 50）。
- `extraOlderVideoCount`：按作者视图下已阅前额外展示数量（默认 1）。
- `authorVideosPageSize`：作者投稿每页数量，控制投稿接口 `ps` 参数默认值（默认 24，最大 42）。
- `authorContinuousExtraPageCount`：作者连续缓存额外页数；连续缓存总目标长度为 `(2 + authorContinuousExtraPageCount) * authorVideosPageSize`（默认 3，最小 1）。
- `authorNonContinuousCachePageCount`：作者非连续块缓存页数上限，按默认 `authorVideosPageSize` 折算容量（默认 3，最小 1）。
- `defaultReadMarkDays`：全局默认近期天数，允许 `1-30` 任意整数，默认 `7`，并作为时间流“至少展示窗口”的基准。
- `enableAllGroup`：是否显示默认“全部”聚合分组（默认开启）。
- `useStorageSync`：是否启用 `chrome.storage.sync`（默认开；超限时回退 local 并提示）。
- 分组已阅时间点写入精度为分钟（秒归零），同一分钟重复写入不新增记录。

## 5. 技术方案

### 5.1 技术栈
- Manifest V3
- Vue 3 + TypeScript
- Vite 构建
- 状态管理：Pinia（或等价轻量方案）

### 5.2 扩展模块划分
- `background`（Service Worker）
  - 统一请求代理（带登录态）
  - 缓存与刷新调度
  - 未读状态计算
- `content`（内容脚本）
  - Header 注入
  - 挂载“分组动态”抽屉 UI
- `options`（设置页）
  - 分组管理
  - 设置项配置
- `shared`（共享层）
  - 类型定义
  - 存储与消息协议
  - API Client

### 5.3 登录态请求策略
- 首选策略：通过扩展发起 `fetch(..., { credentials: 'include' })` 请求 Bilibili 域名接口，使用用户当前登录态 Cookie。
- 权限：声明 `host_permissions` 覆盖 `*.bilibili.com` 必需域名。
- 不主动导出/展示用户 Cookie 原文。
- 对 Bilibili 的 `POST` 写操作，默认在 background 发起前通过 DNR 改写关键来源头，至少覆盖 `Origin`、`Referer`、`Sec-Fetch-Site`，避免请求表现为扩展上下文。
- Content script 负责提供当前页面 `origin/referer`；background 在真正发起单次写请求前临时安装 session-scoped DNR 规则，请求完成后立即清理，并通过后台串行门避免多个写请求并发时串用同一组来源头。
- 若某个 `POST` 写操作在补齐业务参数后仍失败，优先检查该接口是否缺少 DNR 头改写，而不是直接假设 Cookie、CSRF 或请求体字段错误。

### 5.4 Bilibili API（首版拟定）
- 登录与用户信息：用于确认登录态、拿到当前用户 `mid`。
- 我的收藏夹列表：拉取当前用户“创建的收藏夹”。
- 收藏夹资源列表：拉取收藏夹内视频与作者信息。
- 作者投稿列表：按 `mid` 拉取投稿视频（视频数据源）。
- 作者卡片信息：`x/web-interface/card`（作者名称、头像、粉丝数、是否已关注的数据源）。
- 用户关系操作：`x/relation/modify`（关注/取关，需携带 `extend_content`）。
- 视频点赞：`x/web-interface/archive/like`。
- 视频投币：`x/web-interface/coin/add`。

说明：不同接口在 Web 端可能存在签名、分页、风控策略差异；实现阶段将封装可替换 API 适配层，保证后续可替换具体端点而不影响 UI 和核心逻辑。
其中 `POST` 写接口默认视为“需要 DNR 上下文修正”的高风险接口，除非后续验证证明某接口可以稳定不依赖 DNR。

## 7. 交互流程

### 7.1 首次使用
1. 用户安装扩展。
2. 打开设置页，加载“我的收藏夹列表”。
3. 选择收藏夹创建分组，设置别名（可选）。
4. 回到 B 站页面，Header 出现“分组动态”入口。

### 7.2 查看分组动态
1. 点击“分组动态”打开抽屉。
2. 默认选中上次查看条目（开启“全部”分组时可落在 `__bbe_all__`）。
3. 先读取缓存：
   - 分组无缓存：提交收藏夹优先刷新任务并显示“生成中”状态。
   - 分组有缓存但作者缓存不完整：提交缺失作者优先刷新任务，保留已命中的展示内容并显示“正在更新中”。
4. 展示内容并按当前分组有效已阅基线计算红点；不因打开分组自动更新“已读时间”。

### 7.3 混合模式加载更多
1. 初次目标 `timelineMixedMaxCount`。
2. 支持“触底自动加载 + 手动点击加载更多”双触发；仅在仍有更多内容时显示按钮。
3. 每次触发后目标 += 20。
4. 构造器先尝试使用当前缓存；若命中作者缓存边界则仅提交补页意图，不同步等待。
5. 若补页失败，不影响当前缓存结果返回；错误通过状态提示与调试面板可见。
6. 若单轮拉取超过目标，不裁剪。
7. 时间流支持“卡片间点击设已阅”：点击相邻卡片间竖线后，写入该间隙时间点并刷新“上次看到这里”横向分割线。

## 8. 错误与降级
- 未登录：面板显示“请先登录 Bilibili”。
- 接口失败：统一使用通知弹框提示；不以错误文本覆盖主内容区。
- 部分作者拉取失败：记录错误并继续展示已成功部分。
- 若收藏夹刷新结果为空数组（如收藏夹被删除或不可访问）：保持原分组标题与作者列表缓存不变，等待用户在设置页手动删除该分组。
- `storage.sync` 配额超限：自动回退 `storage.local`，提示“已切换本地存储”。

## 9. 权限清单（MV3）
- `storage`
- `alarms`（可选，用于周期性检查）
- `host_permissions`：`https://*.bilibili.com/*`
- `scripting`（如需动态注入）

## 10. 验收标准（首版）
1. 用户可从"我的收藏夹列表"创建/编辑/删除分组。
2. `www` 与 `space` 页面 Header 均出现"分组动态"入口。
3. 面板支持分组切换、时间流/近期投稿/全部投稿模式切换，以及“刷新投稿列表”“刷新收藏夹”两个手动刷新入口。
4. 请求缓存时长可配置，默认 30 分钟。
5. 作者缓存与收藏夹缓存均支持独立后台定时刷新，两个默认间隔均为 10 分钟。
6. 作者级视频缓存跨分组共享，避免重复请求同一 UP 主。
7. 混合模式初始目标默认 50，支持“触底 + 按钮”双触发追加 20，超出目标不裁剪。
8. 作者视图仅在“全部投稿 + 全部”场景下支持作者内分页，分页请求必须携带 `pn + ps`，分页大小由高级设置 `authorVideosPageSize` 控制请求默认值，并优先展示投稿 API 实际返回的 `ps` 参数值，且分页器整体水平居中。
9. 红点按“有效已阅基线”生效：全部投稿筛选（全部/7/14/30 天、10/30 条）不影响分组红点和 Header unread，时间流临时 `全部` 也不影响持久化基线。
10. 设置页“分组列表”每行提供“刷新投稿列表”“刷新收藏夹”两个按钮；抽屉工具栏提供同名按钮，行为保持一致。
11. 设置页支持配置 `schedulerBatchSize`，并对所有调度通道同时生效。
11.1 设置页高级模式支持配置 `authorVideosPageSize`，默认 `24`，最大 `42`。
11.2 设置页高级模式支持配置 `authorContinuousExtraPageCount`，默认 `3`，最小 `1`。
11.3 设置页高级模式支持配置 `authorNonContinuousCachePageCount`，默认 `3`，最小 `1`。
12. 调试页支持“立刻发起调度”；触发后下一次自动调度时间从当前时刻重新起算（`now + interval`）。
13. 调试页支持 Burst 模式监控：可查看 Burst 队列、当前任务（分组名 + 作者名；缺失作者名时回退 MID）、下一次可执行时间，以及是否处于“错误冷却（60s）”。
14. “近期投稿/全部投稿”模式支持“按更新时间倒序”勾选项，默认勾选，且按分组记忆勾选状态。
14.1 未勾选“按更新时间倒序”时，按作者子侧栏支持拖拽重排；拖拽结果按分组持久化，“近期投稿/全部投稿”共享同一份手动顺序，“全部”分组也支持独立手动顺序，并对新增作者采用尾追加策略。
15. 分组已有缓存但作者缓存部分缺失时，前台不清空已展示内容，状态显示“正在更新中”，并自动补齐缺失作者缓存。
16. “刷新投稿列表”会刷新“分组信息（标题、作者列表）+ 作者投稿缓存”，且点击前必须二次确认；“刷新收藏夹”只刷新“分组信息（标题、作者列表）”；若收藏夹返回空数组则不覆盖现有分组缓存。
17. 时间流“加载更多”按钮仅在仍有更多内容时显示；命中作者缓存边界时只提交补页意图，不等待补页完成，列表始终先返回缓存结果。
17.1 作者投稿共享缓存以“连续数组 + 分块候选缓存”建模，不再把 `pn` 直接当成缓存真相；手动翻页、跳页拿到的块会回灌缓存并尝试重建连续数组。
18. 默认提供“全部”聚合分组，且可在设置中关闭显示。
19. “全部”分组只做聚合浏览：不保存独立的分组级近期基线与 `readMark` 历史，其红点继续展示真实分组 unread 的聚合结果。
20. 分组支持“忽略 unread 计数”开关；开启后该分组红点固定为 0。
21. 作者支持“忽略 unread 计数”开关（跨组共享）；开启后该作者视频不参与任何 unread 计算。
22. VideoCard 底部作者信息行右侧提供可点击“已阅对勾”，与“已看完”并存；已阅视频不计入 unread。
23. 按作者标题右侧提供作者级“标记已阅 / 撤销已阅”与“不计算未读”按钮，两者与关注按钮按“左信息组 + 右操作组”布局展示。
24. 作者级已阅时间点跨组共享，且优先级绝对高于分组基线；可通过右键竖线清除。
25. 分组已阅时间点按分钟精度存储；同一分钟重复标记不会新增时间点。
26. 时间流和“近期投稿”共享同一组近期设置：下拉固定提供 `7天内 / 14天内 / 30天内`，若设置页默认值不是三者之一则额外显示该自定义 `N天内`；用户选择的 `N天内` 或具体日期都会作为持久化基线参与 unread。
27. 时间流存在分组“上次看到”时间点时，工具栏显示 `撤销上次看到` 按钮；撤销后会同步影响“近期投稿”模式；“全部”分组不显示该按钮。
27.1 时间流左侧时间线节点支持左键设基线；仅当前带对勾的激活节点支持右键清除当前分组自定义基线，清除后恢复到默认近期范围；“全部”分组不提供这套交互。
28. 作者级已阅标记线显示在两张视频卡片之间的中线位置。
29. 只要处于时间流视图，命中作者级逻辑已阅的视频卡片默认半透明，hover 恢复不透明，并带过渡动画。
30. 时间流“上次看到这里”支持任意已阅时间点，不再仅支持天数分割；分割线显示不截断后方内容。
31. 时间流在任意已阅时间下仍保持“至少当前设置页定义的 N 天窗口（默认 `7天内`）”的展示窗口，避免切换已阅点导致内容突变。
32. 任意接口/操作失败时，前台通过通知弹框提示错误，不再把主内容区替换为错误文本。
33. 关注/取关调用 `x/relation/modify` 时必须携带 `extend_content` 字段；其他扩展参数非强制。
34. 提供可调用的视频点赞与投币 API（消息链路贯通）。
35. 每位作者标题行提供“一键点赞”按钮，可对当前可见视频批量点赞，并通过 `like-action` 通道串行执行。
36. 所有 VideoCard 封面左下角都显示可点击点赞拇指：未点赞为空心白色描边，已点赞显示粉色图标，并支持再次点击取消点赞。
