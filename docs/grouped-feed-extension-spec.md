# 分组动态扩展 Spec（Chrome/Edge + Firefox 本地调试 + Bilibili）

## 1. 背景与目标

### 1.1 背景
Bilibili 现有关注/动态入口信息密集，无法按用户自定义主题分组查看投稿更新。

### 1.2 目标
实现一个 Chrome/Edge 扩展，在用户登录态下请求 Bilibili 接口，允许用户将“自己的收藏夹”定义为分组，并在站点 Header 注入“分组动态”入口，按分组展示收藏夹内作者的投稿视频更新。

### 1.3 非目标
- 不实现桌面通知。
- 不实现跨设备强一致同步（仅依赖浏览器 `storage.sync` 能力，受配额限制）。
- 不支持手动输入收藏夹 ID（首版必须从“我的收藏夹列表”选择）。

## 2. 术语
- 分组：由一个“我的收藏夹”映射而来（1:1）。
- 全部分组：虚拟聚合分组，聚合所有启用中的真实分组，不直接绑定收藏夹。
- 时间流模式：分组内所有作者投稿混排，按发布时间倒序。
- 未观看模式：原作者模式，按作者分段展示，支持作者内分页查看投稿。
- 概览模式：按作者快速浏览，不参与追踪基线计算。
- 追踪已阅时间点：用于 unread 计算（可为具体时间、默认已阅天数或“全部”）。
- 概览筛选：仅影响当前展示、不参与 unread 计算的快速筛选（如 全部/14/30 天、10/30 条）。

## 3. 用户故事
1. 作为登录用户，我可以从“我的收藏夹列表”里选择若干收藏夹，创建分组并设置分组别名。
2. 作为用户，我在 `www.bilibili.com` 与 `space.bilibili.com` 顶部 Header 都能看到“分组动态”入口。
3. 作为用户，我点击入口后看到右侧抽屉面板，并能在左侧分组栏切换不同分组。
4. 作为用户，我可以切换“时间流/未观看/概览”三种视图查看投稿。
5. 作为用户，我可以手动刷新；若距离上次刷新超过阈值（默认 10 分钟），系统自动刷新。
6. 作为用户，我在未查看某分组且出现新投稿时，该入口与对应分组显示红点；红点始终基于已阅时间点计算，不因打开分组自动清除。
7. 作为用户，我在“未观看”模式可勾选“按更新时间倒序”，并记住该分组上次选择状态。
8. 作为用户，我可以使用默认“全部”分组聚合浏览所有分组，并一键“标记全部分组为已阅”。
9. 作为用户，我可以对分组或作者设置“忽略 unread 计数”，避免特定来源影响红点。

## 4. 功能需求

### 4.1 分组管理
- 数据来源：仅“我创建的收藏夹”。
- 创建分组：从收藏夹下拉/列表选择一个收藏夹后创建。
- 约束：一个收藏夹最多绑定一个分组（1:1）。
- 分组字段：
  - `groupId`（扩展内部 ID）
  - `mediaId`（收藏夹 ID）
  - `mediaTitle`（收藏夹原名）
  - `alias`（可选别名）
  - `enabled`（启用状态）
  - `excludeFromUnreadCount`（是否忽略该分组 unread 计数，默认 `false`）
  - `createdAt` / `updatedAt`
- 支持编辑：别名、启用状态、是否忽略 unread 计数。
- 支持删除分组。

#### 4.1.1 默认“全部”分组

- 系统提供虚拟聚合分组“全部”，默认开启（可在设置中关闭）。
- “全部”分组不绑定收藏夹，不写入 `GroupConfig`。
- “全部”分组聚合范围：所有 `enabled === true` 的真实分组。
- “全部”分组工具栏按钮文案为“标记全部分组为已阅”：
  - 点击后对所有启用中的真实分组写入一条当前时间戳的分组已阅记录。
  - 不为“全部”分组本身单独写入已阅记录。

### 4.2 入口注入与面板
- 注入页面：
  - `https://www.bilibili.com/*`
  - `https://space.bilibili.com/*`
- 注入位置：站点顶部 Header 区域，新增“分组动态”入口（视觉风格贴近站点现有导航）。
- 点击行为：打开右侧抽屉面板（内容脚本挂载 Vue 应用）。
- 面板结构：
  - 左侧：分组列表（支持红点）
  - 右侧：当前分组内容区 + 视图模式切换 + 刷新按钮 + 刷新状态（上次刷新时间 / 正在更新中）

#### 4.2.1 Header 入口红点/数字展示

- Header 入口支持两种状态：
  - 红点状态：`unreadCount > 0` 时显示。
  - 数字状态：显示全局 `unreadCount`，`>99` 时显示 `99+`。
- 红点/数字元素复用站点样式类 `red-num--message`，避免自定义视觉与站点风格偏差。
- `unreadCount = 0` 时数字文本为空（隐藏数字），不通过 `display` 切换元素显隐。
- 红点/数字挂载的父节点必须显式设置 `position: relative`，保证绝对定位稳定，不受站点布局变更影响。

#### 4.2.2 抽屉导航状态模型

抽屉侧栏统一使用“导航条目 ID（`activeEntryId`）”驱动，不再使用独立的 `showSettings` / `showDebug` 布尔开关。

- 真实分组条目：使用分组自身 `groupId`。
- 虚拟分组条目（参与 feed 展示）：
  - 全部分组：`__bbe_all__`
- 页面虚拟条目（仅用于页面切换）：
  - 设置页：`__bbe_settings__`
  - 调试页：`__bbe_debug__`

交互规则：
- 点击“全部”后，`activeEntryId = __bbe_all__`，展示聚合后的 feed。
- 点击“设置”后，`activeEntryId = __bbe_settings__`，显示设置页。
- 点击“调试”后，`activeEntryId = __bbe_debug__`，显示调试页。
- 再次点击“设置”或“调试”时，不自动切回分组页（无 toggle 返回行为）。
- 点击任一真实分组后，`activeEntryId = groupId`，显示对应分组内容。

实现约束：
- 当 `activeEntryId` 为真实分组 ID 或 `__bbe_all__` 时，允许触发 feed 请求。
- 设置/调试条目不参与分组缓存、未读统计与分组持久化字段（如 `lastGroupId`）。
- “全部”分组不写入 `GroupConfig`，但可参与 unread 展示与工具栏交互。

### 4.3 投稿聚合规则
- 输入：分组绑定收藏夹中的视频集合。
- 作者提取：从收藏夹视频中提取作者 `mid` 去重。
- 投稿拉取：按作者获取投稿视频列表（仅用于视频列表数据）。
- 作者信息拉取：统一使用 `x/web-interface/card` 获取作者信息（`name`、`face`、`follower`、`following`）。
- 去重：按视频 `bvid` 去重。
- 排序：按投稿发布时间（`pubdate`）倒序。

#### 4.3.1 作者信息来源约束
- 作者名称、头像、粉丝数、关注状态的展示与业务判断，均以 Card API 返回结果为准。
- 仅在“该作者没有任何已缓存 Card 信息”且本轮 Card 请求失败时，允许临时回退使用投稿接口中的作者名兜底展示。
- 一旦该作者已存在 Card 缓存，后续即使 Card 请求失败也必须继续使用缓存，不回退到投稿接口作者信息。

### 4.4 展示模式

#### 4.4.1 时间流模式
- 默认初始目标数量：`50`（可配置项）。
- 滚动增量目标数量：每次触底追加 `20`（固定值，首版不暴露配置）。
- “加载更多”按钮与触底自动加载并存：
  - 当 `hasMoreForMixed = true` 时，列表底部显示“加载更多”按钮。
  - 当 `hasMoreForMixed = false` 时，隐藏“加载更多”按钮，仅显示“没有更多内容了”。
  - 保留触底自动加载行为，两者复用同一 loadMore 链路与同一节流状态。
- 重要规则：
  - 为构造目标数量，可对作者列表执行多轮拉取。
  - 若某轮请求后数量超过目标值，**不裁剪**，全部展示。
  - 仅在“当前可用数据小于目标值”时继续拉取下一轮。

#### 4.4.1.1 时间流按天分段与左侧时间轴
- 时间流列表按“有投稿的自然日”分段，不显示分段标题文案，仅在分段之间保留视觉留白。
- 左侧时间轴与内容滚动联动，按“有投稿日期”渲染节点。
- 时间文案规则：
  - 当天显示 `今天`。
  - 昨天显示 `昨天`。
  - 距离今天 2–7 天（含）显示 `N 天前`（如 `3 天前`）。
  - 其余日期显示 `MM-DD`（不使用“近一周/一周前”等压缩文案）。
- 节点样式状态：
  - 日期分段在可视区域内：显示空心圆节点。
  - 日期分段不在可视区域内：退化为小黑点节点。
- 时间轴压缩窗口规则：
  - 可视区域内的“有投稿日期”节点全部保留，不做压缩。
  - 在可视区域之外，仅额外保留上方 2 个与下方 2 个“有投稿日期”节点。
  - 离屏节点固定停靠在时间轴顶部/底部区域，不贴靠可视节点，避免滚动到临界位置时跳变。
  - 无投稿的日期不占位、不参与前后计数。
  - 不允许无限累积离屏日期节点。

#### 4.4.1.2 时间流“上次看到这里”分割
- 分割线支持任意已阅时间点，不再要求命中自然日 `00:00` 边界。
- 在混排结果中定位第一条“`pubdate < readMarkTs`”的视频，并在该卡片前插入“上次看到这里”横向分割线。
- 若列表首条视频已满足 `pubdate < readMarkTs`，则分割线显示在列表顶部。
- 分割线仅作为视觉分隔，不触发“截断后方旧内容”。

#### 4.4.1.3 时间流展示下界与平滑策略
- 时间流展示必须保留“至少 `defaultReadMarkDays` 天”的可见窗口（默认 7 天）。
- 即使存在分组/作者级已阅时间点，也不按已阅时间硬截断旧内容。
- 展示下界取“最近 N 天窗口下界”与“当前数据最旧可用时间”的平滑组合，避免切换已阅点时出现突变式折叠。

#### 4.4.1.4 时间流“卡片间设置已阅”交互
- 在相邻 VideoCard 之间 hover 时，显示半透明粉色竖线虚影，仅作为交互提示。
- 点击竖线虚影后，将当前分组已阅时间点设置为“两条视频之间”的时间：
  - 常规取值：`olderTs + 1`；
  - 同秒不可切分时退化为 `newerTs`。
- 设置成功后，持久分割统一使用“上次看到这里”横向分割线，不保留竖线作为持久分隔。

#### 4.4.2 未观看模式
- 总量不设上限（由作者数量决定）。
- 每位作者内部按发布时间倒序。
- 每位作者标题右侧展示关注按钮：
  - 未关注：`+ 关注 X万`
  - 已关注：`已关注 X万`
  - `X万` 基于 Card API 的 `follower` 字段格式化。
- 作者标题行布局分为左右两组：
  - 左侧操作组：头像 + 作者名链接 + 关注按钮；
  - 右侧操作组：`标记已阅` + `不计算未读`。
- 点击按钮可直接关注/取消关注（`x/relation/modify`）：
  - 前端先做乐观更新；
  - 后台请求成功后回写最新 Card 信息；
  - 失败时回滚并以通知弹框提示，不替换主内容区。
- 关注请求结构要求：
  - 必须携带 `extend_content`（JSON 字符串：`{"entity":"user","entity_id":fid}`）。
  - 当前实现固定补齐 `gaia_source=web_main`、`spmid=333.1387`、`is_from_frontend_component=true`。
  - 当前实现固定补齐 URL Query：`statistics={"appId":100,"platform":5}` 与 `x-bili-device-req-json={"platform":"web","device":"pc","spmid":"333.1387"}`。
  - `extend_content` 允许经 `URLSearchParams` 编码后提交，不要求保留抓包中的原始冒号字面值。
  - 请求当前由 background 发起，但会在发起前通过 DNR session rule 把 `Origin`、`Referer`、`Sec-Fetch-Site` 改写为贴近页面前端的值。
- 右侧“标记已阅”按钮写入作者级已阅时间点，值为当前时间戳（`now`）。
- “不计数”按钮文案统一改为“`不计算未读`”。
- 工具栏新增勾选项：`按更新时间倒序`（仅在“未观看”模式展示）。
- 勾选项默认值：`true`（默认勾选）。
- 勾选为 `true` 时：
  - 按“该作者在当前筛选条件下可见视频的最新 `pubdate`”倒序排列作者列表。
  - 若某作者在当前筛选条件下无可见视频，则排在列表末尾。
  - 若最新 `pubdate` 相同，按收藏夹作者原始顺序（`authorMids` 顺序）稳定排序。
- 勾选为 `false` 时：作者列表按收藏夹作者原始顺序（`authorMids` 顺序）展示。
- 勾选状态按分组维度持久化记忆，切换分组后恢复各自上次选择。
- 工具栏“已阅”按钮文案：
  - 普通分组：`全部标记已阅`（语义为将当前分组追踪基线推进到 `now`）；
  - “全部”分组：`标记全部分组为已阅`（批量写入真实分组）。

#### 4.4.2.1 作者内分页（仅“全部”筛选）

- 当满足以下任一条件时启用作者内分页：
  - 处于“未观看”模式，且已阅筛选为追踪选项 `全部（t:0）`；
  - 处于“概览”模式，且概览筛选为 `全部`。
- 当处于概览筛选 `N天内 / 14天内 / 30天内 / 10条 / 30条`，或处于非 `t:0` 的追踪筛选时，不显示分页控件，沿用当前筛选结果直出。
- 分页容器（页码区 + 跳页区）在作者分段底部整体水平居中显示。
- 分页大小使用投稿 API 的 `ps` 参数值，不新增设置项覆盖该值。
- 每位作者分页状态独立维护，默认进入第 1 页。
- 分页控件能力：
  - `上一页/下一页`；
  - 页码按钮（页数较多时使用省略号）；
  - 跳页输入（支持直接跳到指定页）。
- 当目标页尚未缓存时：
  - 前台仅提交 `REQUEST_AUTHOR_PAGE` 意图（`groupId + mid + pn`），不等待补页任务完成；
  - 调度器异步执行补页，前台通过后续轮询/下一次读取拿到该页缓存后自动切换到目标页；
  - 若作者已无更多页，返回 `no-more` 并提示用户当前最大可用页。
- 分组切换、已阅筛选切换后，作者分页重置为第 1 页。

#### 4.4.2.2 未观看模式子侧栏导航
- 仅在“未观看”模式显示子侧栏，位置在内容区左侧。
- 子侧栏为竖向列表，不使用时间线样式，不增加额外分段间距。
- 列表项展示：作者头像 + 作者名字（不显示 mid 数字）。
- 点击列表项后，内容区平滑滚动到对应作者分组顶部。
- 子侧栏需要独立滚动；当作者数量较多时，不影响右侧内容区滚动。
- 滚动联动高亮：内容区滚动时，子侧栏自动高亮当前可视区域内的作者。

#### 4.4.2.3 作者“一键点赞”
- 在作者标题左侧操作组（关注按钮同级）新增“一键点赞”按钮。
- 点击后对该作者“当前界面可见视频卡片”执行批量点赞（仅当前筛选 + 当前分页，不隐式拉取未展示页）。
- 批量点赞通过通道化调度器中的 `like-action` 通道串行执行，避免并发请求触发风控。
- 按钮执行中显示 loading 并禁用；批次结束后以通知弹框汇总成功/失败数量。

#### 4.4.3 概览模式（独立于追踪基线）

- 工具栏模式按钮为：`时间流 / 未观看 / 概览`。
- 概览模式使用独立下拉（概览组）：
  - `全部`（概览）
  - `N天内`（如 7 天内，取 `defaultReadMarkDays`）
  - `14天内`
  - `30天内`
  - `10条`
  - `30条`
- 概览模式不显示“标记已阅”按钮。
- 概览组选项语义：
  - 只影响当前列表展示；
  - 不参与分组 unread 与 Header unread 计算；
  - 不写入分组追踪基线（`savedReadMarkTs`）。
- 概览模式下若作者存在作者级已阅时间点：
  - 仅绘制作者级已阅标记线；
  - 不触发“已阅后只保留额外 N 条”的截断逻辑。
- `概览 = 全部` 时：
  - 使用作者全量缓存参与展示；
  - 启用作者内分页；
  - 不受分组已阅时间点与作者级已阅时间点过滤影响。
- `概览 = N天内 / 14天内 / 30天内` 时：
  - 先按对应时间窗口筛选作者视频；
  - 若作者在窗口内命中至少 1 条视频，则只展示命中结果；
  - 若作者在窗口内 0 命中，但缓存中仍存在更旧视频，则补 1 条该作者最新视频作为兜底展示；
  - 兜底展示时在作者标题旁显示“最近更新：X天前”提示；
  - 该兜底视频仅用于概览展示，不改变 unread 计算，也不复用“已阅后额外 N 条”的逻辑。
- `概览 = 10条 / 30条` 时：
  - 仅按条数截断当前作者列表；
  - 不做“保底补 1 条”。
- 只要处于时间流视图，若视频命中“作者级逻辑已阅”（`video.pubdate <= authorReadMarkTs`）：
  - VideoCard 默认半透明；
  - hover 恢复不透明；
  - 不透明度变化带 transition 动画。
- 细节行为以 `docs/read-mark-spec.md` 为准。

#### 4.4.4 作者级已阅标记线定位

- 作者级已阅标记线必须位于“前后两个视频卡片之间”的几何中线。
- 不允许贴靠更新较旧或较新的任一卡片边缘。

#### 4.4.5 VideoCard 已阅对勾

- 在 VideoCard 底部作者信息行右侧新增可点击“已阅对勾”，用于不跳转页面就地标记已阅。

### 4.5 缓存与刷新机制

#### 4.5.1 作者级视频缓存（AuthorVideoCache）

将视频缓存粒度从"按分组"改为"按作者（mid）"，跨分组共享。

全局数据结构 `AuthorVideoCache`，按 `mid` 索引：

```ts
interface AuthorVideoCache {
  mid: number;
  // 作者基础信息统一来自 Card API
  name: string;
  face?: string;
  follower?: number;
  following?: boolean;
  videos: VideoItem[];        // 已拉取的视频，按 pubdate 倒序
  // 每页缓存元数据（页级新鲜度与使用反馈）
  pageState: Record<number, { fetchedAt: number; usedInMixed: boolean; lastUsedAt?: number }>;
  // 该作者当前已缓存的最大页码（至少为 1）
  maxCachedPn: number;
  // 下一次补页请求要拉取的页码（通常为 maxCachedPn + 1）
  nextPn: number;
  hasMore: boolean;           // 是否还有更多页
  // 首页（pn=1）最后一次刷新时间，用于过期判定基线
  firstPageFetchedAt: number;
  // 第二页（pn=2）最后一次预取/刷新时间（可选）
  secondPageFetchedAt?: number;
  lastFetchedAt: number;      // 最后一次从 API 拉取的时间戳（ms）
}
```

存储位置：`chrome.storage.local`（key: `bbe:author-video-cache`）。

缓存过期判定：`Date.now() - lastFetchedAt > refreshIntervalMinutes * 60 * 1000`。

刷新行为：
- 默认刷新首页（`pn=1`）：
  - 过期时优先刷新首页，更新 `firstPageFetchedAt`，并与已有数据合并去重。
  - 首页未过期时，不重复拉首页。
- 页级预取策略（不限于第二页）：
  - 第二页预取是防御性加速，不保证在任意时刻都已存在。
  - 仅当某页数据**实际参与了目标页组构造**时，构造器才回报“该页已被使用”；单纯命中边界不回报。
  - 常规调度仅依据上述“实际使用”回报决定下一页预取。
  - 仅当 `1..K` 页都新鲜且第 `K` 页被标记为“已使用”时，才允许预取第 `K+1` 页。
  - 若第 `K+1` 页虽然已缓存但未被使用，则不会继续预取第 `K+2` 页。
  - 若首页过期，本轮允许暂时保留历史高页缓存，不立即清空；待首页恢复新鲜后再继续页级预取。
- 动态补页策略（`pn>=2`）：
  - 由时间流构造器按需触发，单次每作者最多补 1 页；
  - 若补完仍不足，再进入下一轮“命中边界 -> 补 1 页”。
- 跨页重复视频去重规则：
  - 同一 `bvid` 在多页同时出现时，优先使用 `meta.updatedAt` 更新的一条；
  - 若 `meta.updatedAt` 相同，则使用“来源页新鲜度更高（`pageState[pn].fetchedAt` 更大）”的数据。
- 每次刷新投稿列表时必须同步请求一次 Card API，更新 `name/face/follower/following`（与投稿刷新同频，不单独设 Card 刷新周期）。
- 若本轮 Card 请求失败：
  - 有已有 Card 缓存：保留并继续使用已有缓存；
  - 无 Card 缓存：允许临时使用投稿接口作者名兜底，直到后续 Card 刷新成功。
- 未过期时，直接使用缓存数据。
- 手动刷新（`forceRefresh`）时无视过期判定，强制重新拉取。

#### 4.5.2 分组与作者的关系

分组不再直接持有视频数据，改为持有作者 mid 列表的引用：

```ts
interface GroupFeedCache {
  groupId: string;
  authorMids: number[];       // 该分组包含的作者 mid 列表
  updatedAt: number;          // 作者列表最后更新时间（从收藏夹重建的时间）
}
```

分组刷新流程：
1. 调用 `getAllFavVideos(mediaId)` 获取收藏夹内容，提取作者列表 → 更新 `GroupFeedCache.authorMids`。
2. 对列表中每个 `mid`，检查 `AuthorVideoCache[mid]` 是否过期，过期则刷新。
3. 组装分组视图时，从全局 `AuthorVideoCache` 中按 `authorMids` 聚合数据。

跨分组共享效果：分组 A 刷新了 UP 主 X，分组 B 也包含 UP 主 X 时直接命中缓存。

#### 4.5.3 收藏夹列表缓存

`getAllFavVideos(mediaId)` 的结果也纳入缓存，按 `mediaId` 索引，过期时间同样遵守 `refreshIntervalMinutes`。避免同一分组短时间内重复拉取收藏夹全量数据。

#### 4.5.4 其他请求缓存

| 请求 | 缓存粒度 | 缓存位置 | 过期时间 |
|------|----------|----------|----------|
| `getMyCreatedFolders` | 全局单例 | 内存 | `refreshIntervalMinutes` |
| `getCurrentUser` | 全局单例 | 内存 | `refreshIntervalMinutes` |
| `getWbiKeys` | 全局单例 | 内存 | 2 分钟（固定），遇 412 自动清除重试 |

#### 4.5.5 前台刷新（用户触发）

- 手动刷新（抽屉按钮 / 设置页分组行“立即刷新”）：
  1. 立即向调度器优先入列该分组的收藏夹刷新任务（`group-fav`）。
  2. 收藏夹刷新成功后，基于最新作者列表按缓存状态衔接作者任务：无缓存入 Burst；有缓存且过期入 `author-video` 常规优先；未过期不入队。
  3. 前台通过轮询 `GET_GROUP_FEED` 获取最新缓存，不等待刷新链路同步完成。
- 新增分组（`UPSERT_GROUP` 新建）：
  1. 保存成功后立即向调度器优先入列该分组的 `group-fav` 刷新任务。
  2. 编辑已有分组不自动触发该刷新（避免重复请求）。
- 自动刷新（打开面板或切换分组）：
  1. 若分组无 `GroupFeedCache`：提交 `group-fav` 优先任务，返回 `cacheStatus: 'generating'`。
  2. 若分组已有 `GroupFeedCache`：不自动触发 `group-fav`（收藏夹缓存仅由定时调度和手动刷新驱动）。
  3. 若分组作者缓存仅部分命中：提交缺失作者的 `author-video` 优先任务，返回 `cacheStatus: 'generating'`，并携带当前可展示的缓存结果（不清空已有内容）。
  4. 仅当“本轮需要补齐的作者都完成至少一轮缓存（存在 `AuthorVideoCache[mid].lastFetchedAt`）”后，才返回 `cacheStatus: 'ready'`。
  5. 若当前完全没有可展示内容，则显示“正在生成缓存”；若已有部分内容，则保持内容可见并显示“正在更新中”状态。
- 前台调度边界：
  1. 前台只允许“读缓存 + 提交刷新意图”；可提交 `REQUEST_AUTHOR_PAGE(groupId, mid, pn)` 表达目标页需求，但不能指定调度细节（如队列位置、是否 `failFast`）。
  2. 前台请求不得同步等待调度任务完成；刷新结果通过后续轮询/下一次读取自然可见。

#### 4.5.6 刷新状态展示

- 工具栏右侧状态位规则：
  - `ready`：显示相对时间（如“刚刚刷新 / X 分钟前”）。
  - `generating` 且无可展示内容：显示“正在生成缓存，请稍候...”。
  - `generating` 且已有可展示内容：显示转圈动画 + 文本“正在更新中”，不显示“上次刷新时间”。

#### 4.5.7 后台定时刷新

通过通用调度器 + `chrome.alarms` 实现，Service Worker 休眠后仍可被唤醒。调度细节见 `docs/scheduler-spec.md`。

后台刷新拆分为两条独立通道（互不共享 ratelimit）：
1. `author-video`：刷新作者视频缓存，周期由 `backgroundRefreshIntervalMinutes` 控制。
2. `group-fav`：刷新收藏夹缓存（标题 + 作者列表），周期由 `groupFavRefreshIntervalMinutes` 控制。

批次执行规则保持不变：每个通道都按 `schedulerBatchSize` 分批、批内串行 + 固定间隔、批间按周期均匀分散，并保留失败任务重试机制；详见 `docs/scheduler-spec.md` 3.5 节。
当存在“作者无缓存”目标时触发 Burst 模式：Burst 优先于常规调度，常规任务会在 Burst 清空后恢复；详见 `docs/scheduler-spec.md` 3.6 节。

#### 4.5.8 时间流分页构造与补页

时间流在构造目标片段（如 `1-50`、`51-70`、`71-90`）时遵循以下规则：

1. 先尝试用当前可用缓存直接构造并返回（只保证首页，第二页及以上可能缺失）。
2. 构造过程中若“触及某作者当前缓存最旧一条视频”，且该作者 `hasMore=true`，视为命中分页边界。
3. 命中边界时（包括缺失第二页场景），仅向调度器提交“边界补页意图”；本次读取不等待任务完成。
4. 本次返回继续使用当前缓存 best-effort 构造；补页完成后由下一次读取拿到新结果。
5. 若 Burst 补页失败：记录调度历史并在状态面板可观测，不中止当前缓存读取链路。
6. 构造结果中实际用到的作者页码会回报给调度器，作为后续常规预取的唯一推进信号。
7. 构造时若同一 `bvid` 来自多页候选，按“`meta.updatedAt` 更新优先；若相同则来源页 `fetchedAt` 更新优先”做最终选取。

#### 4.5.9 设置项汇总

| 设置项 | UI 标签 | 含义 | 默认值 | 范围 |
|--------|---------|------|--------|------|
| `refreshIntervalMinutes` | 请求缓存时长（分钟） | API 请求结果的缓存有效期 | 30 | 1–120 |
| `backgroundRefreshIntervalMinutes` | 作者缓存刷新间隔（分钟） | `author-video` 通道 alarm 周期 | 10 | 5–120 |
| `groupFavRefreshIntervalMinutes` | 收藏夹缓存刷新间隔（分钟） | `group-fav` 通道 alarm 周期 | 10 | 5–120 |
| `schedulerBatchSize` | 每批任务数 | 调度器每个通道每批最多执行任务数（全通道共享） | 10 | 1–50 |
| `enableAllGroup` | 显示“全部”分组 | 是否在侧栏显示默认聚合分组 | `true` | `true/false` |

### 4.6 红点（未读）规则
- 分组红点与“追踪基线”绑定，不使用“打开分组即已读”的模型。
- 分组级未读计算：
  - `excludeFromUnreadCount === true` 的分组，未读数固定为 `0`。
  - 概览过滤（全部/N天内/14/30 天、10/30 条）不参与未读计算。
  - 作者若命中“忽略 unread 计数”开关（跨组共享），其视频在所有分组中均不计入 unread。
  - 作者若存在作者级已阅时间点，则该作者使用作者级基线（绝对覆盖分组基线，无关大小）。
  - 其余作者按分组追踪基线（`全部`/`N天内`/历史已阅）计算。
  - 分组未读数 = 分组内去重后视频中同时满足：
    - `pubdate > baselineTs`
    - 且视频最终状态为“未已阅”。
- Header 全局 unread count 采用“按作者聚合”规则：
  - 聚合时忽略被分组开关或作者开关排除的来源；
  - 同一作者跨分组仅按 1 份视频集合去重计数；
  - “全部”分组是聚合视图，不额外放大或改变其他分组基线。

### 4.7 设置项
- `refreshIntervalMinutes`：请求缓存时长（默认 30）。
- `backgroundRefreshIntervalMinutes`：作者缓存刷新间隔（默认 10）。
- `groupFavRefreshIntervalMinutes`：收藏夹缓存刷新间隔（默认 10）。
- `schedulerBatchSize`：调度器每批任务数（默认 10，全通道共享）。
- `timelineMixedMaxCount`：时间流模式目标基数（默认 50）。
- `extraOlderVideoCount`：未观看模式下已阅前额外展示数量（默认 1）。
- `defaultReadMarkDays`：无已阅记录时的默认已阅天数，且作为时间流“至少展示天数”下界（默认 7）。
- `enableAllGroup`：是否显示默认“全部”聚合分组（默认开启）。
- `useStorageSync`：是否启用 `chrome.storage.sync`（默认开；超限时回退 local 并提示）。
- 分组已阅时间点写入精度为分钟（秒归零），同一分钟重复写入不新增记录。

## 5. 技术方案

### 5.1 技术栈
- Manifest V3
- Vue 3 + TypeScript
- Vite 构建
- 状态管理：Pinia（或等价轻量方案）

### 5.2 扩展模块划分
- `background`（Service Worker）
  - 统一请求代理（带登录态）
  - 缓存与刷新调度
  - 未读状态计算
- `content`（内容脚本）
  - Header 注入
  - 挂载“分组动态”抽屉 UI
- `options`（设置页）
  - 分组管理
  - 设置项配置
- `shared`（共享层）
  - 类型定义
  - 存储与消息协议
  - API Client

### 5.3 登录态请求策略
- 首选策略：通过扩展发起 `fetch(..., { credentials: 'include' })` 请求 Bilibili 域名接口，使用用户当前登录态 Cookie。
- 权限：声明 `host_permissions` 覆盖 `*.bilibili.com` 必需域名。
- 不主动导出/展示用户 Cookie 原文。
- 对 Bilibili 的 `POST` 写操作，默认在 background 发起前通过 DNR 改写关键来源头，至少覆盖 `Origin`、`Referer`、`Sec-Fetch-Site`，避免请求表现为扩展上下文。
- Content script 负责提供当前页面 `origin/referer`；background 负责结合 `sender.tab.id` 安装 session-scoped、tab-scoped 的 DNR 规则，避免多个标签页串用同一组来源头。
- 若某个 `POST` 写操作在补齐业务参数后仍失败，优先检查该接口是否缺少 DNR 头改写，而不是直接假设 Cookie、CSRF 或请求体字段错误。

### 5.4 Bilibili API（首版拟定）
- 登录与用户信息：用于确认登录态、拿到当前用户 `mid`。
- 我的收藏夹列表：拉取当前用户“创建的收藏夹”。
- 收藏夹资源列表：拉取收藏夹内视频与作者信息。
- 作者投稿列表：按 `mid` 拉取投稿视频（视频数据源）。
- 作者卡片信息：`x/web-interface/card`（作者名称、头像、粉丝数、是否已关注的数据源）。
- 用户关系操作：`x/relation/modify`（关注/取关，需携带 `extend_content`）。
- 视频点赞：`x/web-interface/archive/like`。
- 视频投币：`x/web-interface/coin/add`。

说明：不同接口在 Web 端可能存在签名、分页、风控策略差异；实现阶段将封装可替换 API 适配层，保证后续可替换具体端点而不影响 UI 和核心逻辑。
其中 `POST` 写接口默认视为“需要 DNR 上下文修正”的高风险接口，除非后续验证证明某接口可以稳定不依赖 DNR。

## 7. 交互流程

### 7.1 首次使用
1. 用户安装扩展。
2. 打开设置页，加载“我的收藏夹列表”。
3. 选择收藏夹创建分组，设置别名（可选）。
4. 回到 B 站页面，Header 出现“分组动态”入口。

### 7.2 查看分组动态
1. 点击“分组动态”打开抽屉。
2. 默认选中上次查看条目（开启“全部”分组时可落在 `__bbe_all__`）。
3. 先读取缓存：
   - 分组无缓存：提交收藏夹优先刷新任务并显示“生成中”状态。
   - 分组有缓存但作者缓存不完整：提交缺失作者优先刷新任务，保留已命中的展示内容并显示“正在更新中”。
4. 展示内容并按当前分组已阅时间点计算红点；不因打开分组自动更新“已读时间”。

### 7.3 混合模式加载更多
1. 初次目标 `timelineMixedMaxCount`。
2. 支持“触底自动加载 + 手动点击加载更多”双触发；仅在仍有更多内容时显示按钮。
3. 每次触发后目标 += 20。
4. 构造器先尝试使用当前缓存；若命中作者缓存边界则仅提交补页意图，不同步等待。
5. 若补页失败，不影响当前缓存结果返回；错误通过状态提示与调试面板可见。
6. 若单轮拉取超过目标，不裁剪。
7. 时间流支持“卡片间点击设已阅”：点击相邻卡片间竖线后，写入该间隙时间点并刷新“上次看到这里”横向分割线。

## 8. 错误与降级
- 未登录：面板显示“请先登录 Bilibili”。
- 接口失败：统一使用通知弹框提示；不以错误文本覆盖主内容区。
- 部分作者拉取失败：记录错误并继续展示已成功部分。
- 若收藏夹刷新结果为空数组（如收藏夹被删除或不可访问）：保持原分组标题与作者列表缓存不变，等待用户在设置页手动删除该分组。
- `storage.sync` 配额超限：自动回退 `storage.local`，提示“已切换本地存储”。

## 9. 权限清单（MV3）
- `storage`
- `alarms`（可选，用于周期性检查）
- `host_permissions`：`https://*.bilibili.com/*`
- `scripting`（如需动态注入）

## 10. 验收标准（首版）
1. 用户可从"我的收藏夹列表"创建/编辑/删除分组。
2. `www` 与 `space` 页面 Header 均出现"分组动态"入口。
3. 面板支持分组切换、时间流/未观看/概览模式切换、手动刷新。
4. 请求缓存时长可配置，默认 30 分钟。
5. 作者缓存与收藏夹缓存均支持独立后台定时刷新，两个默认间隔均为 10 分钟。
6. 作者级视频缓存跨分组共享，避免重复请求同一 UP 主。
7. 混合模式初始目标默认 50，支持“触底 + 按钮”双触发追加 20，超出目标不裁剪。
8. 作者视图在“未观看 + 全部（t:0）”与“概览 + 全部”两种场景下均支持作者内分页，分页大小使用投稿 API 的 `ps` 参数值，且分页器整体水平居中。
9. 红点按“追踪基线”生效：概览过滤（全部/N天内/14/30 天、10/30 条）不影响分组红点和 Header unread。
10. 设置页“分组列表”每行提供“立即刷新”按钮，行为与抽屉手动刷新一致。
11. 设置页支持配置 `schedulerBatchSize`，并对所有调度通道同时生效。
12. 调试页支持“立刻发起调度”；触发后下一次自动调度时间从当前时刻重新起算（`now + interval`）。
13. 调试页支持 Burst 模式监控：可查看 Burst 队列、当前任务（分组名 + 作者名；缺失作者名时回退 MID）、下一次可执行时间，以及是否处于“错误冷却（60s）”。
14. “未观看”模式支持“按更新时间倒序”勾选项，默认勾选，且按分组记忆勾选状态。
15. 分组已有缓存但作者缓存部分缺失时，前台不清空已展示内容，状态显示“正在更新中”，并自动补齐缺失作者缓存。
16. 手动刷新会刷新“分组信息（标题、作者列表）+ 作者投稿缓存”；若收藏夹返回空数组则不覆盖现有分组缓存。
17. 时间流“加载更多”按钮仅在仍有更多内容时显示；命中作者缓存边界时只提交补页意图，不等待补页完成，列表始终先返回缓存结果。
18. 默认提供“全部”聚合分组，且可在设置中关闭显示。
19. “全部”分组工具栏支持“标记全部分组为已阅”，会批量写入所有启用分组的已阅时间点。
20. 分组支持“忽略 unread 计数”开关；开启后该分组红点固定为 0。
21. 作者支持“忽略 unread 计数”开关（跨组共享）；开启后该作者视频不参与任何 unread 计算。
22. VideoCard 底部作者信息行右侧提供可点击“已阅对勾”，与“已看完”并存；已阅视频不计入 unread。
23. 按作者标题右侧提供“标记已阅（作者级写 now）”与“不计算未读”按钮，两者与关注按钮按“左信息组 + 右操作组”布局展示。
24. 作者级已阅时间点跨组共享，且优先级绝对高于分组基线；可通过右键竖线清除。
25. 分组已阅时间点按分钟精度存储；同一分钟重复标记不会新增时间点。
26. 概览模式下作者级已阅仅影响标记线展示，不参与列表过滤，也不触发“额外 N 条”截断。
27. 概览模式在 `N天内 / 14天内 / 30天内` 下，当作者时间窗内无命中视频但存在更旧缓存时，仍保底展示 1 条最新视频，并显示“最近更新：X天前”。
28. 作者级已阅标记线显示在两张视频卡片之间的中线位置。
29. 只要处于时间流视图，命中作者级逻辑已阅的视频卡片默认半透明，hover 恢复不透明，并带过渡动画。
30. 时间流“上次看到这里”支持任意已阅时间点，不再仅支持天数分割；分割线显示不截断后方内容。
31. 时间流在任意已阅时间下仍保持“至少 `defaultReadMarkDays` 天（默认 7 天）”展示窗口，避免切换已阅点导致内容突变。
32. 任意接口/操作失败时，前台通过通知弹框提示错误，不再把主内容区替换为错误文本。
33. 关注/取关调用 `x/relation/modify` 时必须携带 `extend_content` 字段；其他扩展参数非强制。
34. 提供可调用的视频点赞与投币 API（消息链路贯通）。
35. 每位作者标题行提供“一键点赞”按钮，可对当前可见视频批量点赞，并通过 `like-action` 通道串行执行。
36. 所有 VideoCard 封面左下角都显示可点击点赞拇指：未点赞为空心白色描边，已点赞显示粉色图标，并支持再次点击取消点赞。
